## 2. 総評

**小規模チームの成果物として明確に水準以上。** 設計判断がコードコメントに「WHY+自己批判」付きで残り、脅威モデル文書・契約テスト(IDOR、認可シグナルの静的スキャン、Node+SQLite での移植性実証)・冪等マイグレーション・キルスイッチ(墓標方式)まで揃っている。

「ホストは顧客データに到達経路を持たない」「pull 型更新・失敗時は現行版維持」「破壊的操作はツール自体を提供しない」という統治原則が、文書上の建前ではなく実装と一致している点は高く評価できる。具体的には:

- **暗号設計**: HKDF による用途別サブ鍵分離(api-keys / member-pii / files / session)、AES-256-GCM、Ed25519 ライセンス署名は「署名した実バイト列を搬送して検証」する KMS 分離耐性のある形式
- **Webhook 防御**: LINE 署名検証、Stripe 署名の timing-safe 比較、deploy_url の SSRF バリデーション
- **エージェント統制**: 対外・破壊系操作の承認キュー、`requiredRole` による道具単位の認可、オートパイロットは opt-in かつ削除/force-push/権限変更の道具を構造的に持たない設計
- **移植性の実証**: Ports & Adapters が宣言だけでなく、同一コア/パーツを Node+SQLite で走らせる適合性テストにより CF 非依存を回帰検証している

ただし、**本番商用化のゲートとすべき矛盾・欠陥が数点ある。** 以下、重大度順。

---

## 3. 重大(本番提供前に必須)

### 3-1. 顧客環境で MASTER_KEY 保護ゲートが一度も発火しない

`apps/client/src/lib/client.ts` の `resolveMasterKey` は `ENVIRONMENT === "production"` のときのみ KV 自動生成をブロックする。しかし**配布テンプレートの `wrangler.jsonc` には `ENVIRONMENT` が設定されていない**(vars は `HOST_BASE_URL` のみ)。

結果、設計書 §10.1 の「鍵と暗号文の同居禁止」ゲートは**自社の env.production だけを守り、実際の顧客デプロイは全件、暗号鍵が暗号文と同一の LICENSE KV に同居したまま恒久運用**される。アカウント侵害時にアプリ層暗号化(APIキー・PII・ファイル)が無力化する、コード内で自ら警告している状態そのものが顧客のデフォルトになっている。

README は「秘密鍵などの設定は不要」を訴求しており、ゼロ設定というプロダクト方針とセキュリティ要件が正面衝突している。対応はどちらかに倒すこと:

1. 「顧客はゼロ設定= KV 鍵保管を許容する」と明示的に決定し、規約・説明資料にリスクとして文書化する
2. デプロイ/アクティベーションフローに Worker Secret 投入を組み込む(自動化の余地あり)

現状は**最も守るべき相手にだけ、最強の防御が効いていない**。

### 3-2. 更新チェーンのトラストアンカーが自己宣言(TOFU)

`prebuild-update.mjs`(日和見ローダ)は、リリース検証用公開鍵を **tarball の配信を告知するのと同じホストから取得**する(`/api/release/pubkey`、フォールバックで `/api/pubkey`)。

これでは署名検証は実質「ホストがそう言った」以上の保証を持たず、TLS と等価。ホスト Worker またはその Secret(`RELEASE_SIGNING_JWK` 含む)が侵害されると、**署名済み悪性バンドルが全顧客の Worker に自動展開され、顧客の D1/KV/APIキーへ到達する**。自己ホストモデルの最大の売り(当社は顧客データに触れない)が、更新チェーン一点で崩れる構図。

**対策**: リリース検証用公開鍵を配布リポに同梱(ピン留め)する。更新自体が署名されるため鍵ファイルは更新で運搬でき、ローテーションは旧鍵によるクロス署名で対応可能。修正コストは小さく、効果は大きい。client 側 `getVerifyJwk`(ライセンス検証鍵)も同型の TOFU だが、こちらはホストがライセンス発行者そのものなので影響は限定的。

### 3-3. セッションの失効不能(最大 7 日)

`apps/client/src/lib/auth.ts` のセッションはステートレス HMAC Cookie(7 日)で、role が内包され、個別失効の手段がない。

会員除名・権限降格・admin 解任が**最大 7 日間反映されない**経路が残る。LINE webhook は都度 `identity.memberOf` で active 会員か確認しており良い実装だが、Cookie 認証の API 群は `ses.role` を信用している。会計・名簿・PII を扱う組織向けツールとしては実害のあるギャップ。

**対策**: 短 TTL +スライド更新時の DB 照合、またはセッションテーブル化(D1/KV)による即時失効。

## 4. 中程度

### 4-1. マイグレーションランナーの構造的脆さ

`migrate.ts` の SQL 分割は「`;` 分割+ `--` 行末除去」の素朴なパーサで、**トリガー定義や `;` / `--` を含む文字列リテラルを持つ将来の SQL で静かに壊れる**。2026-06-04 の Cron 全停止の教訓(bare "duplicate" 無視の廃止)がコメントに残っている割に、構造的には同種の落とし穴が残存。

また初回同時リクエストでの並行適用にレース排他がない。DDL は冪等エラー無視で救えるが、INSERT 系マイグレーションは二重適用し得る。

**対策**: KV ベースの best-effort ロック+「マイグレーションは DDL 限定(データ投入はアプリ層)」の規約明文化。

### 4-2. 公開サイト XSS の防御が自作サニタイザ一枚

`SitePublic.astro` の CSP も `script-src 'unsafe-inline'` のため、管理者入力 HTML(公開 LP)に対する防御は `sanitize.ts` 単独。実体デコード+ allowlist 方式は旧正規表現方式より明確に優れるが、自作サニタイザは mXSS・パーサ差分系のバイパス史が長い領域。

**対策**: せめて公開 LP ページだけは nonce ベース CSP へ移行し多層化する。「管理者入力だから低リスク」は、複数 role 運用(clerical 等)では成立しない場面がある点にも留意。

### 4-3. deploy_code の公開露出窓

throwaway 公開リポの `report.json` に deploy_code が平文で置かれ、host の `/api/deploy-report` は first-write-wins。GitHub をスキャンする攻撃者が正規デプロイより先に偽 URL を POST すると、**正規導入が詰まる(オンボーディング DoS)**。ライセンストークンは Google ログイン突合がなければ取れないため情報窃取には至らないが、deploy_url は後段でホストがサーバーサイド fetch する宛先(A2A)にもなる。

**対策**: Google ログイン突合での確定(§2.7 経路)を正とし、deploy-report は仮登録扱いにする。

## 5. 軽微・衛生

- **文書ドリフト**: `CLAUDE.md` の `SCHEMA_VERSION = 20` は実コード(25)と乖離。誤操作防止のために毎セッション読み込む運用文書こそ、最新性が生命線
- **重複実装**: `verifyStripeSig` が host(`lib/billing.ts`)と client(`api/site/stripe-webhook.ts`)で重複。セキュリティクリティカルなコードは shared へ一本化を(乖離した瞬間が事故の温床)
- **トークンのログ残留**: `/api/check` が GET クエリにライセンストークンを載せる。observability 有効=ログに残る。ヘッダまたは POST ボディへ
- **静的認可テストの限界**: `api-authz.contract.test.ts` は「`getSession(` 等の文字列存在」検査であり、全分岐での enforcement の証明ではない。回帰網としては優秀だが、過信しないこと(files の IDOR ランタイムテストの方式を他ルートにも展開する価値あり)
- **「難読化」の実態**: 公開バンドルは標準の minify 済み Astro ビルドで、ロジックは復元可能。配布形態としては問題ないが、IP 保護としては期待しないこと(社内文書の「難読化済み」表現は期待値を誤らせる)
- **手動デプロイの footgun**: client の本番デプロイで `--env production` を付け忘れると別アカウント向け設定に向かう件、CLAUDE.md の警告より `deploy:prod` スクリプト化でエラークラスごと消す方が確実
- **ホットパスのオーバーヘッド**: middleware で全リクエストに `ensureSchema`(KV read)+ `bootCheck` が走る。軽量とはいえ、`/api/` 含む全経路の固定コスト。許容範囲だが認識しておくこと

## 6. 評価が高かった点(維持すべき設計)

- pull 型更新+「検証 NG /取得失敗→現行版維持」のフェイルセーフ原則が実装レベルで貫徹されている
- アクティベーション完了後の throwaway リポ即削除など、露出最小化の運用が随所に組み込まれている
- 自己修復ループ(client →アウトボックス→ host → Issue 化→人+ Claude)の責務分界が明確(「集積・通知まで」がシステムの責務)
- プラン(契約)とエンタイトルメント(実効権限)の二層分離は、nonprofit 審査と Stripe 入金が並行する現実に即した正しい判断
- `entitlementForGate` の鮮度ゲート(解約後の Pro 残存窓の縮小)など、エッジケースへの目配りが細かい
