# セキュリティポリシー / 脆弱性報告

合同会社貘（以下「当社」）は baku-office の安全性を重視しています。脆弱性を発見された場合は、以下の方針に従ってご報告ください。

## 報告窓口

- メール：**baku@baku-llc.co.jp**（件名に `[security]` を付けてください）
- 公開連絡先：本リポジトリ配信アプリの `/.well-known/security.txt` も参照できます。

報告は日本語または英語で受け付けます。

## 報告に含めていただきたい情報

- 影響を受ける箇所（URL・エンドポイント・コンポーネント／host か client か）
- 再現手順（PoC・リクエスト例。可能なら最小限の再現）
- 想定される影響（情報漏えい・認可バイパス・改ざん 等）
- 発見に用いた環境（バージョン／コミット、ブラウザ等）

## 本製品の構成（報告前にご理解ください）

baku-office は**クライアント自己ホスト型**です。アプリ（client）は各利用団体自身の Cloudflare アカウント上で稼働し、業務データ・暗号鍵はその団体のアカウント内で完結します。当社（host）はライセンス発行・配信・課金のみを担い、利用団体の業務データに到達する経路を構造的に持ちません。

このため脆弱性は次のいずれかに分類されます。報告時に区別いただけると対応が早まります。

- **製品コードの脆弱性**（当社が修正・配布で対処）：認可バイパス、IDOR、CSRF、XSS、更新チェーンの検証不備、暗号処理の欠陥 等。
- **個別団体の設定・運用に起因する事象**（当該団体の責任範囲）：当該団体の Cloudflare アカウント設定、投入した Secret の管理、独自に追加したコード 等。

## 対応方針（目安）

- **受領確認**：原則 3 営業日以内。
- **初期トリアージ**：影響度評価を 7 営業日以内を目標。
- **修正・配布**：重大度に応じて優先。高リスクは速やかに署名リリースで配布し、利用団体へ通知します。
- **協調的開示**：修正配布前の公開はお控えください。修正完了後、報告者と調整のうえ開示します。

## 対象範囲

**対象内**：認可・認証境界、IDOR、CSRF、公開LPの XSS、更新チェーン（署名検証・トラストアンカー）、暗号処理、レート制限、機密情報の露出。

**対象外**：

- 当社の制御外にある第三者サービス（Cloudflare／Stripe／Google／LINE 等）の問題。各提供元へご報告ください。
- DoS／DDoS、物理攻撃、ソーシャルエンジニアリング、スパム送信。
- 利用団体自身の設定ミス・認証情報の管理不備に起因する事象。
- 実害の伴わない理論上の指摘（バナー情報、未使用ヘッダの欠如 等）のみ。

## 謝辞

有効なご報告には謝意を表し、ご希望に応じて修正の公表時に報告者として記載します（協調的開示の遵守が前提）。**現時点で金銭的報奨金（バグバウンティ）の制度はありません。**
