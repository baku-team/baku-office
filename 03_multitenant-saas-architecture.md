# 03. マルチテナント SaaS アーキテクチャ（1000社スケール・CLI運用前提）

[01](01_deploy-pipeline.md) / [02](02_cloud-agent.md) のキットを、**多数の企業・団体へ提供し運用を当社が担う事業**として成立させる設計。**新運用形態＝「顧客所有アカウント＋当社ホストの承認・ロジック配信＋ライセンス制御（本文は非中継）」をベース**とし、当社は原則データを保有せず、導入支援・共同運用で報酬を得る。すべての運用は CLI（`tenantctl`）から回す。

## 0. 設計の前提（このドキュメントの確定事項）

- **運用モデル（ベース）**: **顧客所有アカウント**（Cloudflare/Anthropic/LINE は顧客契約）でエージェントが稼働。**Anthropic/LINE 等は顧客が直接やり取り**し、**顧客データ（本文）は当社サーバを一切通らない（非中継）**。当社コントロールプレーンは**承認（ライセンス）＋ロジック配信のみ**。**多リージョン・アクティブ-アクティブ**で全社SPOFを回避。詳細は 17章。
- **IP保護・依存（意図的）**: 中核は**固定Worker（事前配備・難読化）＋ライセンスゲートで配信する config（データ）**。コードのエフェメラル配信はWorkersのeval禁止で不可＝**事前配備が前提**、エフェメラルは設定のみ（17-2）。**無ライセンス＝config配信停止＝エージェント機能のみ停止／アカウント・データは無停止**。**顧客のデータは人質にしない**（lock-in は機能依存で担保）。IP保護のモートは「コード秘匿」でなく承認ゲート＋更新＋蓄積＋ネットワーク。
- **オンボーディング**: 顧客がアカウントを所有。初回は**顧客自身が申込（規約準拠）＋当社が文書/動画/同席で支援**（負担軽減）。完全な作成代行は Cloudflare の書面許可が要（17-6・18-1）。
- **課金/原価**: インフラ・AI・LINE は**顧客契約＝当社COGSほぼゼロ**。報酬は**初期構築費＋月額管理費**＝**粗利ほぼ純利**（5章）。
- **提供スコープ（現時点）**: **BYOK（顧客アカウント型）のみ**。AI込み/当社ホスト型は**将来オプション・現状非提供**（当社が処理委託先に戻り越境/DPAが再発するため見送り）。
- **当社ホスト型（将来オプション・現状非提供）**: 自前アカウントを持ちたくない小規模向けに、当社アカウントのプール/シャードで簡易提供する案（この場合のみ当社がデータ保有・2章）。現時点では提供しない。
- **コード**: 当社が非公開で完全管理。顧客環境には**配信ロジックをローカル実行するエージェントWorker**のみ（難読化＋ライセンス契約）。
- **隔離**: 顧客アカウント運用は**1社で完結（最大の分離）**。爆発半径＝1社。
- **セキュリティ・オプション（任意）**: 暗号化／PIIマスキング／データリージョン選択（8・16章）。機微・国内限定要件のある社が選択。
- **バックアップ**: 顧客アカウント＋別トラストドメイン（別プロバイダ/ローカル）へ暗号化退避。本番提供前に必須化（10章）。
- **エージェント**: **Claude 主＋Workers AI 委譲**（[02](02_cloud-agent.md)）。AI は顧客契約（BYOK相当）が基本。利用者別コストを可視化し上限で警告→停止→増枠（5-3）。
- **運用**: GitOps ＋ 単一コントロールCLI（`tenantctl`）。**当社は常設アクセスを持たず**、日常（課金・監視・config更新・自己修復）は**クライアント自走で完全自動**。初回構築・復旧のみ顧客が**時限JIT付与→失効**（17-5）。
- **常時監視**: 決定論ガード（常時）＋ Workers AI 巡回 ＋ Claude 異常時の三段（9章）。
- **ポータビリティ**: DB/ストレージは Cloudflare 非依存（S3互換API＋素のSQL＋アダプタ層・12章）。
- **運用4原則（恒久）**：(1) **当社アカウント事由でも顧客は止まらない**＝クライアントは最後の正常リース＋設定で**自走**、当社CPは更新/enforcement専用＋CF外セカンダリ（17-3/17-4）。(2) **導入時を除き人手なしの自動運用**（承認・課金停止・監視は自動、例外のみ最小人手・17-11）。(3) **無料枠から逸脱しない**（キャッシュ/サンプリング/D1カウンタでusage overageを出さない。固定費は規模拡大時のWorkers Paid $5のみ）。(4) **すべて CLI（`tenantctl`）で操作**。

## 1. 全体構成：承認・配信のみの当社CP ／ 直接やり取りの顧客アカウント

```
┌─ 当社コントロールプレーン（多リージョン・アクティブ-アクティブ）──────┐
│  ライセンス/承認サーバ … サブスク状態でリース発行/停止             │
│  config配信 … プロンプト/ツール/人格/パラメータ（データ）を         │
│              ライセンスゲートで配信（コードは事前配備・配信しない）  │
│  レジストリ・監査・tenantctl（共同管理）                           │
│  ※顧客データ（本文）は当社を一切通らない                           │
└───────────────┬────────────────────────────────────────────────────┘
   ①ライセンス/設定 要求 → ②署名リース＋エフェメラル設定（本文なし）
                ▼
┌─ 顧客所有アカウント（データ・処理・連携はすべて顧客側）────────────┐
│  エージェントWorker … 受領した設定でローカル実行                   │
│    └ Anthropic / LINE / Discord を【顧客が直接】呼ぶ（本文は当社不経由） │
│  D1 / R2 … 顧客のデータ at rest（当社は保有しない）                 │
└────────────────────────────────────────────────────────────────────┘
```

- **当社は承認とロジック配信のみ**。**顧客データ（本文）は当社サーバを一切通らない（非中継）**。Anthropic/LINE は**顧客が直接**契約・通信し、当社はキーも本文も持たない。
- **可用性**：1リージョン停止→他リージョンが即時継続（顧客無影響）。Cloudflare 全体障害は**契約で免責**し、**ローカルサーバーへのバックアップ（10-4）を推奨**。
- **キルスイッチ**：無ライセンス＝**エージェント機能のみ停止**。顧客のアカウント・データは無停止（17-3）。
- **爆発半径＝1社**。**当社ホスト型（予算オプション）**は当社アカウントのプール/シャード（2章）で簡易提供（この場合のみ当社がデータ保有）。

## 2. シャーディング設計（当社ホスト型＝予算オプション）

> **【現時点では非提供・将来オプション】** この章は**当社ホスト型**（顧客が自前アカウントを持たない簡易プラン）専用の設計で、**現提供は BYOK のみ**のため当面は使わない。**ベースは顧客所有アカウント運用（1章・17章）**で、1社1アカウントのためシャーディングは不要。以下は将来「当社アカウントに小規模多数を相乗りさせる」場合にのみ適用し、この時だけ当社がデータを保有する。

### 2-1. シャードの単位と容量

| 項目 | 値 |
|---|---|
| 1シャード | 1 Cloudflareアカウント |
| 収容上限 | **100社/シャード** |
| 増設契機 | 現行シャードが100社に達したら次シャードを作成 |
| 1000社時 | シャード約10個＋専用数個 → 基本料 約$50/月＋専用分 |

> **利点**: アカウント作成は「100社ごとに1回」と稀。大量自動作成（Tenant Platform API）が不要で、手動/半自動で足りる。デプロイも約10アカウントへの matrix で済み、社数比例にならない。

### 2-2. テナントのシャード割当とルーティング

- レジストリが `tenant_id → shard_id（= account_id）` を保持。
- **LINE Webhook はオンボーディング時にそのシャードの Worker URL を直接登録**（例: `https://shard-003.<sub>.workers.dev/line/webhook`）。中央ルーターを置かないので単一障害点を作らない。
- シャード内では LINE `destination` → `tenant_id` を引いて、共有Worker が社を識別。

### 2-6. Free → Paid 透過スケール（ユーザー無影響）

- **Workers Free → Paid の昇格は課金トグルのみ**。URL・バインド・コードは不変＝**再デプロイ不要・ダウンタイムなし・LINE Webhook URLも不変**。Durable Objects と高い上限が解放される。
- **シャード（=アカウント）単位で独立昇格**。混雑したシャードだけ Paid へ、新シャードは Free 開始でよい。
- 上限カウンタは `RateLimiter` インターフェース裏で **D1(無料) → DO(Paid)** を差し替え（カウンタは月次リセットなので**月境界で切替えれば移行が単純**）。インターフェース経由のためエンドポイント変更なし。
- 昇格契機の検知: `tenantctl` がシャードの Free 枠（リクエスト/日・D1書込/日）逼迫を検出して通知。

### 2-3. 専用アカウント（重要案件）

- 停止が許されないサービス・機密案件は `isolation_tier = dedicated` とし、**1社1アカウント**。
- 同じ共有コードをデプロイ。課金・データ・ToS執行が完全分離。
- 管理は同じ `tenantctl` / GitOps 経由（個別運用だが手順は共通）。

### 2-4. 隔離ティアまとめ

| ティア | 配置 | D1 | R2 | 隔離強度 | 対象 |
|---|---|---|---|---|---|
| standard（既定） | シャード相乗り | プールD1＋tenant_id | 共有バケット＋prefix | 論理 | 大多数 |
| isolated | シャード内 | 社別D1 | 社別prefix/バケット | 物理(DB) | 分離要件あり |
| dedicated | 専用アカウント | 専用D1 | 専用バケット | 完全分離 | 重要/機密案件 |

### 2-5. ティア別の検閲・バックアップ方針

| ティア | 取込検閲 | バックアップ | 理由 |
|---|---|---|---|
| standard / isolated（共有シャード） | **必須**（取込関門・6-3） | 別ドメインへ日次（全社共通） | 同居100社を1社の違反から守るため検閲は外せない |
| dedicated（専用アカウント） | **なし** | **専用の別バックアップを個別に確保** | 機密性優先（内容を読まない）＋ 隔離済みで爆発半径=1社。リスクは自社責任で受容し、復旧はバックアップで担保 |

> 検閲は「共有インフラ上の同居者を守る」ための措置。同居者がいない専用アカウントでは便益が消え、逆に機密内容を読むデメリットが勝つため実施しない。dedicated は暗号化（8章）併用時は内容検閲が原理的に不能でもある。

### 2-7. テナント内のメンバースコープ（個人/共有）

隔離の境界は **会社（テナント）** であり、**メンバー単位ではない**。DB・ストレージは**テナント単位で管理**し、メンバー間の分離は**アプリ層の論理スコープ**で行う。

**DB/ストレージの管理単位**
- DB（D1）・ストレージ（R2）は**会社ごと**（standard＝プールD1＋`tenant_id`／isolated・dedicated＝専用）。**メンバーごとには分けない**。
- 理由：同じ会社のメンバーは共有データ（ナレッジ・共有文書）を使うため共通ストアが必要。メンバー別の物理分割は非現実的。
- メンバー分離は**行レベル**：各レコードに `owner_id`（メンバー）＋ `scope`（personal/team/company）を持たせ、クエリヘルパで強制。R2は `tenant_id/owner_id/...` パス＋scopeメタで管理。

**スコープと参照可否**

| スコープ | 例 | 参照できる人 |
|---|---|---|
| **personal**（既定） | 個人タスク・マイツール・会話履歴 | 本人のみ |
| **team** | チーム/プロジェクトの成果物 | 同チーム＋役職権限者 |
| **company** | 社内ナレッジ・共有文書 | 社内の権限保持者 |

- **既定は personal**。社員Aの個人作業を社員Bが勝手に見ることはできない。
- Aが成果物を team/company に**明示共有**、または管理者/役職権限で付与すれば B も参照可。
- **会社（テナント）間は常に不可**。横断は構造的に遮断。
- 参照・共有・権限変更は `audit_log` に記録。break-glass（運営の緊急アクセス）も時限・監査つき。

> まとめ：物理的な管理単位は「**会社**」。その中の「**個人/共有**」はアプリ層のスコープ（`owner_id`＋`scope`）と役職権限で制御する。

## 4. CLI運用（GitOps ＋ tenantctl）

テナントを宣言的マニフェストで管理し、CLIがリコンサイル。Claude Code がこのCLIを操作する。

```bash
# プロビジョニング・設定（冪等・--json）
tenantctl shard create                          # 新シャード(アカウント)を準備・登録
tenantctl tenant apply tenants/acme.yaml        # 作成/更新（空きシャードへ自動割当）
tenantctl tenant list --json
tenantctl flags set acme feature.broadcast=on
tenantctl cap set acme --tokens 2000000 --cost-jpy 5000 --storage-gb 5
tenantctl cap topup acme --add-jpy 3000 --until 2026-06-30   # 上限枠の一時増加（追加課金）
tenantctl sub set acme --status active --renews 2026-07-01
tenantctl byok set acme --provider anthropic
tenantctl promote acme --tier dedicated         # 重要案件を専用アカウントへ昇格

# デプロイ・マイグレーション
tenantctl deploy                                # 全シャードへ（約10アカウント matrix）
tenantctl migrate --tier isolated               # 物理D1社だけ一括

# 運用・インシデント
tenantctl suspend acme --reason "ToS違反"        # キルスイッチ（即時停止）
tenantctl breakglass issue acme --scope read --ttl 1h --reason "障害調査"
tenantctl content quarantine <key>              # 違反物の即時隔離
tenantctl usage report --month 2026-06 --by-member --json   # 利用者別の使用量/円コスト
tenantctl monitor status --json                 # 監視ヘルス/アラート
tenantctl restore acme --d1-time-travel 2026-06-01T00:00:00Z
```

- **宣言的＝GitOps**: `tenants/*.yaml` がソース・オブ・トゥルース。`apply` で差分のみ反映。CI(GitHub Actions)で自動化も可。
- **冪等＆再開可能**: 中断地点から再開（`provisioning_jobs` パターン）。
- **機械可読**: `--json` で Claude Code が結果を解析→次操作へ連鎖。
- **破壊的操作は確認**＋全件 `audit_log` 記録。

## 5. サブスク・課金・使用量上限

### 5-0. 2つの課金モデル

| | **顧客アカウント型（ベース）** | 当社ホスト型（予算オプション） |
|---|---|---|
| データ保有 | 顧客 | 当社 |
| インフラ/AI/LINE 費 | **顧客契約（当社COGS外）** | 当社が負担し料金に内包 |
| 当社の報酬 | **初期構築費 ＋ 月額管理費**（＋任意のアカウント作成支援費） | 月額サブスク（AI込み） |
| 粗利益率 | **ほぼ純利（COGSほぼゼロ）** | 約95%（5-1 試算） |
| 稼働条件 | **当社ライセンスリース**（未払いで即停止・17章） | サブスク状態でゲート（5-2） |

- **【現時点の提供は 顧客アカウント型（BYOK）のみ】**。当社ホスト型（AI込み）は**将来オプション・現状非提供**（当社が処理委託先に戻り越境/DPAが再発するため見送り）。
- ベースは顧客アカウント型。当社はインフラ/AI/LINE費を負担しないため、原価はほぼゼロ＝報酬の大半が利益。AI実費は顧客がAnthropicへ直接（BYOK）。
- 初期導入の負担軽減として、**顧客自身の申込を当社が有償で支援**（**ガイド付きBYOK**＝設定支援で簡単さとデータ非保有を両立）。完全代行はCF書面許可が要（17-6）。
- 以下 5-1 の月額・粗利試算は**当社ホスト型（将来）**のもの。**現提供の顧客アカウント型は「構築費＋月額管理費」**（AI/インフラは顧客負担）。

### 5-1. プラン設計（機能バンドル）

プラン＝ `feature_flags ＋ caps ＋ 隔離ティア ＋ バックアップ方針 ＋ 暗号化` のバンドル。レジストリで定義し `tenantctl plan set <id> <plan>` で適用。プラン変更は設定反映のみ（デプロイ不要）、隔離ティア変更時のみ `promote`/`migrate` が走る。

| プラン例 | 個別管理(隔離) | 暗号化 | バックアップ | 検閲 | 想定 |
|---|---|---|---|---|---|
| **Basic** | 共有シャード(論理分離) | なし | なし/簡易 | 取込関門 | 大多数 |
| **Secure** | isolated D1 | ON（エンベロープ） | クラウド日次（別ドメイン） | 取込関門 | 分離・機密要件 |
| **Dedicated** | 専用アカウント | ON | クラウド＋**ローカル**(契約同意) | なし（機密のため） | 重要/機密案件 |

- セキュリティ（暗号化・モデレーション強度）・バックアップ（無/クラウド/ローカル/頻度）・個別管理（隔離ティア）を**プランの差別化軸**として段階販売。
- 上限（トークン/コスト/ストレージ/アップロード）もプランに紐付け。
- **ユーザー定義ツール**（13章）もプラン軸：Basic＝マクロ/リマインダー、Secure〜＝＋外部APIコネクタ。チャネル選択（LINE/Discord等）は全プラン共通。
- **任意オプション（プラン/テナント単位でON/OFF）**：PIIマスキング（16-1）／データリージョン選択（16-2・dedicated向け）／暗号化（8章）。機微・国内限定要件のある社が選択。

**月額（税別）と採算設計（粗利益率95%以上を目標）**

| プラン | 月額 | 内包AI枠(原価) | 推定原価(AI＋基盤) | 粗利益率 |
|---|---|---|---|---|
| Basic | ¥9,800 | 〜¥430 | 約¥470 | 約95.2% |
| Secure | ¥29,800 | 〜¥1,350 | 約¥1,470 | 約95.1% |
| Dedicated | ¥120,000〜 | 〜¥4,800 | 約¥5,800 | 約95.2% |

- 原価の中心は **AI実費（Claude API）のみ**。インフラはシャード共有で1社あたり実質誤差、LINE/Discord配信費は顧客側負担（当社COGS外）。
- **AI原価を料金の5%以内**に収める内包枠設計で粗利益率95%以上を確保。担保策：①ハード上限（5-3）②BYOKで大口の実費を顧客負担化 ③モデル3段＋Workers AI委譲＋プロンプトキャッシュで単価圧縮 ④共有インフラ。
- 内包AI枠＝月間AI使用上限（原価ベース、`cap set --cost-jpy` と連動）。¥150/$・Claude標準単価・Sonnet既定で試算。
- **AI課金の住み分け**：**標準プラン＝AI込み**（実費の約20倍を内包＝粗利95%）／**大口＝BYOK**（AI別・顧客がAnthropicへ実費直払い）。大口は内包枠だと割高・上限に当たりやすいため、BYOKが顧客・当社双方に合理的（当社はAI原価が外れ粗利が上振れ）。BYOK割引プランの設定も可。
- 別途：決済手数料（カード約3.6%）は取引コスト（年額/振込で軽減）。値上げ・円安はプラン改定で吸収。

### 5-2. ゲーティング・課金・ライフサイクル

- **ゲーティング**: リクエスト毎に `status == active` ＆ `subscription_renews_at` を確認（KVキャッシュ）。停止中は機能ロック。
- **BYOK（Bring Your Own Key）**: 顧客が **Anthropic と直接契約**して自社APIキーを取得し、**顧客のエージェントWorkerが直接呼び出す（当社は中継しない）**。**AI実費は顧客↔Anthropic の直接課金**となり当社COGSから外れる（＝当社の粗利はほぼ100%に近づく）。設定はCLI/管理画面からのみ、値は末尾マスク・Secrets保管・ログ非出力。AIコスト上限は緩和（暴走防止のレート制限は維持）。顧客側は自社アカウントの利用ティア/レート上限の確保が必要。使用量・コストは引き続き当社画面で可視化。
- **ライフサイクル**: 解約 → 猶予 → エクスポート → 削除 を cron ジョブで自動化（`provisioning_jobs.step_progress` で冪等管理）。

### 5-2b. マルチプロバイダBYOK（能力レジストリ・必須ブレイン・能力対応）

AIを「**能力（capability）**」で抽象化し、各能力に**任意のプロバイダをBYOK**で割り当てる。すべて**顧客の鍵・顧客契約・非中継**で、当社COGS≒0を維持。

- **メインブレインは必須**：`chat` 能力に **Claude／GPT／Gemini 等から1つ**を設定（オーケストレーター）。これが無いと稼働しない。
- **能力タイプ（例）**：`chat`(ブレイン)／`image_gen`／`video_gen`／`transcribe`(話者分離)／`tts`／`embed`／`web_search`／`ocr`。各能力に**プロバイダを割当**（例 transcribe=Gemini/Deepgram、image_gen=OpenAI/Stability、video_gen=Runway/Veo）。
- **能力対応オーケストレーション**：ブレインには**設定済み能力だけがツールとして見える**。必要な能力が **設定済み→BYOKで実行**／**未設定→「未設定です。有効化しますか？」と案内**、または **Workers AI 無料フォールバック**（embed・基本transcribe(Whisper)・基本image・ocr など対応分のみ）。
- **実装（17-2と整合）**：事前配備の**汎用インタープリタWorkerが各能力のアダプタを内蔵**。`config` で「**どの能力に・どのプロバイダ・どの鍵ref**」を配信（コードでなくデータ）。鍵は**顧客Secrets**、登録時に**疎通検証**＋末尾マスク＋当社非保持。
- **データモデル**：`providers`(tenant_id, capability, provider, key_ref, endpoint?, status, is_brain)。**必須制約**：`capability=chat` かつ `is_brain=true` が1つ。
- **CLI**：`tenantctl brain set <id> --provider gemini`／`provider set <id> --capability video_gen --provider runway`（鍵は顧客が入力・検証）／`provider list <id> --json`。
- **規約/コスト**：各プロバイダの Usage Policy 順守は配信ロジックで担保（例：高リスク領域はアドバイス限定・AI開示）。実費は顧客のBYOK契約＝顧客負担。能力別の使用量も可視化（5-3）。
- 例：**音声議事録**＝`transcribe`(Gemini/Deepgram で話者分離)＋`chat`(要約)。**画像/動画制作**＝`image_gen`/`video_gen` を設定した団体のみ有効。使わない団体は未設定のまま（コスト/UIゼロ）。
- **要望ドリブンの拡張**：「◯◯したい」に対し能力を足して拡張できる。手軽さ3段階＝**①既存能力タイプをBYOK設定（即時・自己サービス）／②ユーザー定義の外部APIコネクタ（13-2・コード不要・ガード内）／③新カテゴリの一級アダプタ追加（当社が段階配信17-9）**。ブレインは**未設定の要望を検知して「有効化しますか？」と案内→ウィザード(17-6b)でBYOK登録→即利用**、または Workers AI 無料フォールバックを提案。新プロバイダのアカウント作成は顧客自身（代理不可）。
- **オーケストレーション・プロンプトに案内を組込**：system prompt に「**利用可能な能力・スキル・プラグイン一覧（configから注入）**」＋「**未設定の要望は"できません"で終わらせず、必要な能力を説明し有効化の道筋（ウィザード/外部コネクタ/無料フォールバック/当社へのリクエスト）と代替案を案内する**」指示を埋め込む（02のSYSTEM_PROMPT）。能力の増減はconfig配信で即反映。

### 5-3. コスト可視化・上限ライフサイクル（AIは Claude 主・従量）

**可視化（誰が・いくら）**
- AI呼び出し毎に `tenant_id / member_id / model / 入出力トークン / 円換算コスト` を記録。
- 明細は **Workers Analytics Engine**（大量・安価）、集計は **D1ロールアップ**（社別・**利用者別**の月次）。
- 円換算 = モデル単価表 × USD/JPYレート。`tenantctl usage report --tenant X --by-member --json` で即確認。

**上限ライフサイクル（警告 → 停止 → 増枠）** — 呼び出し**前**に月次コスト上限(円)をゲート判定：

| 状態 | 動作 |
|---|---|
| 80% / 90% 到達 | **アラート**（提供者＋任意で団体管理者）。`tenantctl alert` |
| 100% 到達 | **稼働停止**（Claude停止）。ポリシーで「完全停止＋増枠案内」/「Workers AIのみへデグレード」を選択 |
| 一時増枠 | **追加課金で上限枠を一時拡大**。`tenantctl cap topup <id> --add-jpy 3000 --until 2026-06-30`（有効期限付きで加算） |

- 出力トークンは事前確定しないため、ゲートは「**月累計 ≧ 上限なら新規呼び出しをブロック**」で効かせ、実行中リクエストの超過は `max_tokens` で画す。
- カウンタ実装は `RateLimiter` 経由（無料時 D1 / Paid時 DO・2-6）。

## 6. アンチアビュース（不正利用・不正アクセス・違反アップロード）

「起きる前提」で多層防御。詳細な脅威別対策は本章に集約。

### 6-1. リソース乱用・課金被害
- ストレージ/アップロード/DB書込/AIコストに**社別ハード上限**（DO）。超過は拒否・デグレード。
- アップロード: 最大サイズ・件数/日・拡張子/MIME allowlist。
- エッジ: WAF・Rate Limiting・Bot Management・Turnstile。

### 6-2. 不正アクセス
- 横断不可を構造で担保（クエリヘルパが `tenant_id` 強制、生UPDATE/DELETE無し、R2は prefix 限定の短命署名URL）。
- R2は**常にprivate**・公開URL不発行。
- 社別スコープのR2トークン（アカウント全体キーを各社処理に渡さない）。
- 常設god-mode無し → break-glass（時限・スコープ・理由・監査）のみ。
- 鍵ローテーション、漏洩疑い時即時無効化。

### 6-3. 違反・違法アップロード（＋ ToS爆発半径）→ 7章と連動

**対象は共有シャード（standard / isolated）のみ。dedicated は検閲なし（2-5）。**
R2分離は管理コスト過大のため不採用とし、**取込検閲で発生確率そのものを下げる**方針（影響低減はシャーディングが担当）。

**取込関門（合格物のみ本保存）**

```
アップロード → quarantine領域(本保存しない)
   ├ ① サイズ/件数上限・拡張子allowlist
   ├ ② マジックバイトでMIME検証（拡張子偽装を弾く）
   ├ ③ マルウェアスキャン（キュー、完了まで隔離）
   ├ ④ CSAMハッシュ照合（NCMEC/PhotoDNA系）
   ├ ⑤ 画像モデレーション（Workers AI vision / 外部API）
   └ ⑥ 文書はテキスト抽出→モデレーション（任意）
        │ 全合格 → 本領域へ昇格（private＋短命署名URLのみ）
        └ 不合格 → 即削除＋フラグ＋(必要に応じ)社の凍結・通報
```

- private厳守・公開URL不発行で拡散・インデックス・通報の温床を断つ。
- 新規/低信頼テナントはアップロードを絞る（レピュテーション制）。フラグ品は人手レビューキューへ。
- スキャン判定を `audit_log` に記録（"善意のモデレーション"の証跡＝ToS執行の抑止材料）。
- **自前の高速takedown＋通報体制（DMCA代理人・NCMEC）**。Enterprise の是正猶予に頼れない前提なので、Cloudflareより先に隔離・削除できる速度が重要。

> dedicated（検閲なし）でも、Cloudflareのプラットフォーム側ベースライン検知は適用され得る。専用顧客には**自社コンテンツの責任を契約（利用規約/AUP）で明確化**しておくこと。

## 7. ToS執行リスクの封じ込め

Cloudflareの規約執行は**アカウント単位**。重大違反はアカウント停止/解約に至り得るため、爆発半径をシャードで限定する。

**当面 Enterprise契約（是正猶予）は前提にしない**。代わりに「確率を下げる（検閲）＋影響を限定（シャード）＋確実に復旧（バックアップ）」の3点で受け止める。

| 対策 | 効果 | 状態 |
|---|---|---|
| シャーディング（100社/アカウント） | 1アカウント停止でも被害は≤100社 | 採用 |
| 取込関門で合格物のみ永続化（共有シャード） | 違反物をそもそもCloudflareに残さない | 採用（6-3） |
| 重要案件は専用アカウント | 停止が許されない社を完全分離（爆発半径=1社） | 採用 |
| 自前の高速takedown | Cloudflareより先に隔離・削除 | 採用 |
| **別トラストドメインへのバックアップ＋復旧** | **アカウント停止からの別アカウント復旧（生命線）** | **必須（10章）** |
| Enterprise/事業者契約 | notice-and-cure（是正猶予） | 将来検討（現状未取得前提） |
| R2ストレージの別アカウント分離 | シャード停止時にcompute生存 | **不採用**（管理コスト過大・便益限定。検閲強化で代替） |

> **生命線はバックアップ**: 是正猶予がない以上、シャードが突然停止しても**バックアップから別アカウントへ復旧できること**が事業継続の要。停止は「起きたら≤100社・即復旧」へ落とし込む。

## 8. 暗号化（オプション）

標準では Cloudflare が保存時 AES-256（管理鍵）で暗号化済み。**「本人以外は復元不可」を満たしたい社/データのみ**、アプリ層のエンベロープ暗号化を追加で有効化する。

- **方式**: 保存前に Worker 内で社別 DEK（AES-GCM）暗号化 → 保存先は暗号文のみ。DEK は外部KMS/別アカウントSecrets Storeで封印、必要時のみ開封・永続化しない。R2は SSE-C で顧客鍵をリクエスト毎に渡す。
- **効果**: D1ダンプ/R2オブジェクトを抜かれても暗号文のみ。鍵を別途破らない限り復元不可。社別なので波及しない。
- **データ3分類で監視と両立**（次章）。

### データ3分類

| 分類 | 暗号化 | エージェントの扱い |
|---|---|---|
| 機微（連絡先・名簿） | 社別鍵で強暗号 | 触らない（メタデータのみ） |
| 業務データ（処理対象） | エンベロープ暗号化、Worker内で一時復号のみ・即破棄・監査 | 必要時だけ復号 |
| ファイル（R2） | SSE-C＋取込時検査→暗号化 | アップロード関門で検査、以後暗号文 |

> 構造上、稼働中Workerは処理の瞬間だけ平文に触れる（完全ゼロ知識は server側処理と両立しない）。復号は最小化＋監査で対処。

## 9. 常時監視エージェント（三段構成・基本無料枠内）

> **新モデルでの整合**：顧客アカウント運用では監視は**顧客アカウント側で実行**し、当社へは**メタデータ/アラートのみ**（本文は非中継）。当社ホスト型（予算オプション）でのみ当社が直接監視する。

### 9-0. 監視のメタデータ化（本文を見ずに監視する）

当社は本文を保有しないため、監視は**数値・signal・ラベルのみ**で行う。

| 当社が受け取る（メタデータ） | 当社が受け取らない |
|---|---|
| リクエスト数・**トークン数（数値）**・モデル・推定コスト・時刻 | メッセージ本文 |
| エラー率・レイテンシ・HTTPステータス・例外の**種別と件数** | ファイル内容 |
| アビューズsignal（「スキャン拒否」「上限到達」「ACK逸脱」を**カテゴリ＋件数＋深刻度**） | AIの入出力本文 |
| 監査イベント（利用者ID・ツール名・成功/失敗・時刻） | 個人情報の中身 |

- **顧客側エージェントが「本文を含まない構造化イベント」だけを送信**（Workers Analytics Engine 等）。スキーマに本文フィールドを持たせず、送信前lint/検査で混入防止。IDは擬名化可。
- 当社の監視三段（決定論→Workers AI巡回→Claude異常時）は**このメタデータ列**に対して走る。
- **本文が要る検査（コンテンツモデレーション等）は顧客側で実行**し、当社へは**判定結果（clean/flagged＋カテゴリ）のみ**を送る。
- 対処は**ライセンス/設定の制御**（上限縮小・機能停止）で本文を見ずに完結。
- **限界（トレードオフ）**：本文を見ないため「**回答の質・本文中の微妙な漏れ**」など内容依存の問題は中央検知できない → 顧客側ガード＋顧客報告で拾う。
- コスト：Analytics Engine は無料枠大・低単価。サンプリング/集約で**$0〜低十$/月**。

```
[1) 決定論ガード]  常時・全リクエスト（安価）
   DOレート上限 / クォータ判定 / WAF / 署名検証 / tenant_idスコープ
        │ 逸脱イベント
        ▼
[2) Workers AI 巡回]  Cron（数分毎）＋ R2/監査のイベント駆動
   使用量スパイク・書込洪水・アクセス異常・モデレーション結果・
   break-glass濫用・監査ログの不審シーケンスを走査（暗号化下でもメタデータは平文）
        │ 重要度高
        ▼
[3) Claude トリアージ]  異常時のみ
   文脈判断 → 自動対処（throttle/quarantine/suspend）＋ アラート
```

- 入力: Workers Analytics Engine（メトリクス）＋ `audit_log`（D1）＋ R2イベント通知。
- 全リクエストにClaudeを当てない（コスト）。高頻度は決定論＋Workers AIで吸収し、Claudeは例外時のみ。
- CLI: `tenantctl monitor status / rules set / alert tail / anomaly investigate`。

## 10. バックアップ・事業継続（生命線・全ティア必須）

Enterprise の是正猶予がない前提なので、**「別トラストドメインへのバックアップ＋別アカウント復旧」が事業継続の最重要施策**。

> **導入タイミング**: テスト運用中は後回し可。ただし**他社への提供（本番）を開始する前に必須化**する。

### 10-0. 退避先の選択（別Cloudflareアカウント / 別プロバイダ）

| 退避先 | 管理 | 守れる脅威 | 位置づけ |
|---|---|---|---|
| 別Cloudflareアカウント | 楽（同一ツール/モデル） | per-shardのToS停止（最頻） | **まずはこれで十分**。ただし同一プロバイダ＝相関リスク |
| 別プロバイダ（B2/S3等） | やや増（別IAM/別請求/コピージョブ。S3互換＋R2はegress無料でコピーは安価） | 上記＋Cloudflare全体停止・顧客関係打ち切り・全体障害 | 全面依存を切りたい／規模拡大時 |
| **ローカルサーバー**（契約同意者向け） | 増（オンプレ運用・回線/保全は顧客/自社責任） | 上記＋クラウド全面障害。手元保管 | **Dedicated等のオプション**。データがクラウド外へ出るため**契約で明示同意必須** |

- **どちらでも「暗号化して退避」が肝**：退避先は暗号文のみを持つため、退避先のセキュリティ水準に結果が左右されにくい（ベンダーロックインも避けられる→12章）。
- 鍵は最小権限（できれば追記専用）＋**オブジェクトロック/Write-Once**で改ざん・削除を防止。鍵ローテ・MFA・アクセスログ。
- **ローカルは Pull型推奨**：ローカル側がS3互換APIでクラウドのバックアップを取りに行く（MinIO同期 or rclone）。クラウドからローカル内部への着信を開けず安全。
- バックアップ着手時（テスト運用後）に提供規模・プランを見て退避先を確定する。

### 10-1. アカウント内機能とアカウント外バックアップを分ける

| 用途 | 機能 | 守れる脅威 | 守れない脅威 |
|---|---|---|---|
| 誤削除・改ざんの巻き戻し | D1 **Time Travel**(最大30日PITR) / R2 **バージョニング** | 操作ミス・破壊 | **アカウント停止/喪失**（機能ごと消える） |
| 停止/喪失からの復旧 | **アカウント外バックアップ** | アカウント停止・解約・侵害 | — |

→ アカウント内機能だけでは停止に耐えられないため、**必ずアカウント外（別トラストドメイン）へ退避**する。

### 10-2. バックアップ仕様

- **退避先**: 本番とは**別のCloudflareアカウント、または別プロバイダ**（B2/S3等）。本番停止と道連れにしない。
- **対象**: D1 はエクスポート（SQL/論理ダンプ）を外部へ／R2 はオブジェクトを外部バケットへコピー／中央レジストリ・`tenants/*.yaml` は Git（Cloudflare外）に既存。
- **頻度**: 共有シャードは**日次**。dedicated（重要案件）は**より高頻度＋専用の別バックアップ先を個別確保**（2-5）。
- **暗号化との両立**: 暗号化データは暗号文のまま退避（退避先は平文を見ない）。**鍵は別途・安全に（KMS/別ドメイン）バックアップ**しないと復元不能になる点に注意。
- **検証**: 復旧は「新規アカウントへ丸ごと戻す」操作になるため、**定期的にリストア訓練**を行う。

### 10-3. 復旧・インシデント手順（CLIランブック）

```bash
tenantctl backup run --shard shard-003          # 手動バックアップ
tenantctl backup verify --shard shard-003       # 退避先の整合性確認
tenantctl restore --shard shard-003 --to <new-account>   # 別アカウントへ丸ごと復旧
tenantctl restore acme --d1-time-travel <ts>    # 軽微な巻き戻し（アカウント内）
```

- インシデント手順: 検知 → `suspend` → スナップショット → `breakglass`調査 → 是正／必要なら `restore --to` で別アカウント復旧。

### 10-4. ローカルサーバーのハウジング（場所貸し・任意/上位）

顧客所有のサーバーを**当社事務所に設置（ハウジング/コロケーション）**し、暗号化バックアップを退避する。当社は**場所・電源・回線・物理セキュリティを提供**し、**データの保有・閲覧はしない**（場所貸しの立場）。

- **技術**：顧客所有ハード＋暗号化（**顧客が鍵を保有**）。当社は**論理アクセスを持たない**（施錠/封印ラック・入退室監査）。バックアップは pull 型で、当社からは中身を読めない。国内事務所なら**国内データ所在**も満たす。
- **法的位置づけ（要顧問弁護士確認）**：顧客がハード所有・鍵保有・暗号化済み・当社は論理アクセスなし、であれば当社は**「場所貸し（ハウジング）＋設備運用」**の立場で、個人データの「取扱い」には当たらない可能性。**データ非保有の方針をさらに徹底**できる。
- **責任・要件**：物理セキュリティ（入退室管理・施錠・防火・電源/回線冗長）、保険、**寄託としての善管注意義務**。ハード保守は remote hands。盗難時も暗号化で被害最小だが物理インシデントとして扱う。
- **位置づけ**：物理サーバーゆえ大規模スケールには不向き。**国内保管・物理管理を求める上位/機密案件向けオプション**（dedicated と併用）。

## 12. ベンダーロックイン回避（データ層ポータビリティ）

DB/ストレージを Cloudflare 非依存にし、将来 R2/D1/DO を別実装へ載せ替え可能にする。**アダプタ層は後付けが難しいため最初から導入**する。

### 12-1. 抽象化の方針

| 層 | 方針 | 既定実装 | 差し替え先 |
|---|---|---|---|
| ストレージ | **S3互換APIに統一**。R2を「S3実装の一つ」として扱う | R2 | AWS S3 / Backblaze B2 / Wasabi / 自前MinIO |
| DB | **素のSQL＋リポジトリ(アダプタ)層**。移植可能なマイグレーションSQL。D1固有API(Time Travel等)に業務ロジックを依存させない | D1 | SQLite / Turso(libSQL) / Postgres / ローカル |
| KV・キャッシュ | `KVStore` インターフェース | CF KV | Redis互換 |
| カウンタ・レート | `RateLimiter`/`Counter` インターフェース | Durable Objects | Redis(atomic INCR) 等 |
| 推論 | 用途指定（`summarize`/`classify`等）でルーティング | Workers AI / Claude API | 他プロバイダ |

### 12-2. 注意点・トレードオフ

- **最も剥がしにくいのは Durable Objects**（原子カウンタ・状態調整）。代替実装インターフェース（Redis等）を最初から用意。
- 計算層(Workers)は本対象外だが、**Web標準API中心で記述しCF固有バインドをアダプタ裏に隔離**すれば、将来 Node/別エッジ runtime への移行余地が残る。
- 抽象層のぶん実装が一段増え、CFネイティブの利便（DOの手軽さ・CF内egress無料の最適化）を一部諦める。**DB/ストレージのポータビリティ確保が目的**で、ゼロコストではない点を許容する。
- バックアップ・ローカル退避（10章）は、この S3互換＋SQLエクスポートのポータビリティがあって初めて素直に成立する。

## 13. ユーザー定義ツール・スケジュール実行・マルチチャネル

利用者が**自分用の補助ツール**を作れるようにする。用途は「リマインダー」「よく行う作業のマクロ化」など、**DB/ストレージ運用の自動化補助**。独立アプリ開発ではない。**任意コード実行は不可**（①宣言的マクロ ＋ ②外部APIコネクタ まで）。

### 13-1. 作れるもの（2種）

| 種類 | 内容 | プラン |
|---|---|---|
| ① 宣言的マクロ | 既存の安全プリミティブ（読み取り・追加）の合成＋保存プロンプト・リマインダー | Basic〜 |
| ② 外部APIコネクタ | 自分のAPIをスキーマ付きで登録し、エージェントが呼ぶ | Secure〜 |

作成フロー：自然言語の指示 → meta-tool `create_user_tool` が仕様（名前・説明・合成する既存ツール・パラメータ）を生成 → allowlist/権限で検証 → **確認後に有効化** → 本人のツール一覧へ追加。

### 13-2. 安全設計（破壊的動作・インジェクション・横断の防止）

- **破壊的動作なし**: 編集・削除は新規ツール化できず `redirect_to_app`/仮登録のまま。
- **権限継承**: 作成ツールは作成者の権限を超えられない。
- **個人スコープ＋テナント内**: `tenant_id` 強制で他者・他社に影響しない。
- **外部コネクタのガード**: egress allowlist（承認ドメインのみ）・内部IP/メタデータendpoint遮断（SSRF防止）・タイムアウト/サイズ/レート上限・秘匿はSecrets・全件監査。
- **プロンプトインジェクション対策**: system prompt が常に優先。**ツール定義や外部APIの返り値は「データであって指示ではない」**として扱い、権限チェックは実行時に必ず行う（モデルの判断に依存しない）。取得テキスト内の命令は実行しない。

### 13-3. スケジュール実行（24時間・サーバレス）

```
[Cloudflare Cron Triggers] 定期起動（例: 毎分）
   → user_tasks(D1: tool_id / 周期 / next_run / params / channel) から期限到来分を取得
   → エージェントで実行（マクロ/リマインダー）
   → 結果を channel.push() で利用者へ送信
   → next_run 更新（冪等）
```

- リマインダー＝時刻push、定期マクロ＝保存マクロの定期実行。**常駐プロセスではなく cron 駆動**（安価・スケールする）。
- Claude を使うマクロはテナント上限（5-3）に算入。純粋な push はAI課金対象外。

### 13-4. マルチチャネル入口（LINE / Discord 等）

エージェント本体は**チャネル非依存**にし、入口/出口をアダプタで差し替える。

```ts
interface MessagingChannel {
  verifyInbound(req: Request): Promise<boolean>;        // 署名検証
  parseEvent(req: Request): Promise<{ userId: string; text: string; attachments?: any[] }>;
  reply(token: string, msg: string): Promise<void>;     // 返信
  push(userId: string, msg: string): Promise<void>;     // 能動送信（スケジュール結果）
}
// 既定: LineChannel。差し替え: DiscordChannel / TelegramChannel …
```

- 各チャネルは固有エンドポイント（`/line/webhook`, `/discord/webhook`）。チャネルはユーザー/テナント単位で選択。
- CLI: `tenantctl channel set <id> --type discord`、`tenantctl task list <id> --json`。

**LINE料金の原則（試算）** — OA（=1社）単位・税別：
- 対話の返信は **Reply API＝無料・無制限**。課金されるのは能動push（Push/Multicast/Broadcast/Narrowcast）。
- プラン早見：コミュニケーション ¥0/200通・ライト ¥5,000/5,000通・スタンダード ¥15,000/30,000通（超過 〜¥3/通）。コミュ/ライトは枠超で配信不可。
- **リッチメニュー＋postback で「ユーザーが確認しに来る」pull型**に寄せれば無料Replyで返せる。push主体・高頻度は **Discord（bot無料）** も選択可。
- ※2026/10/1 にLINE追加メッセージ料金改定予定（要再確認）。

**チャネル選択の指針** — オンボーディング時に想定push頻度をヒアリングして提案：

| 利用スタイル | 推奨チャネル | 理由 |
|---|---|---|
| 対話中心・pushは少なめ | **LINE** | Reply無料。利用者が既にLINEにいて導入が容易 |
| **LINEコストを抑えたい** | **Discord** | push課金なし。LINEの月額(¥5,000〜)・通数枠を回避 |
| **pushを積極活用**（高頻度の通知・日報・アラート） | **Discord** | bot無料でpush実質無制限。LINEだとスタンダード¥15,000＋従量で高額化しやすい |

- 既定提案ロジック：**「push主体／コスト最優先」なら Discord、「LINEリーチ重視／push軽量」なら LINE**。チャネルは `MessagingChannel` 抽象でユーザー/テナント単位に選択・**後から切替可**。
- LINEのpush残枠（13-6）が逼迫した利用者には、エージェントが Discord 併用/移行を案内できる。

### 13-5. 所要時間に応じた Reply/確認分岐（無料Replyで完結）

`replyToken` は約1分・1回のみ有効。1分超の生成・実行は完了Pushになりがちなので、**所要時間を計測・予測して pull型に逃がす**。

- **計測**: タスク実行の開始〜終了を記録し、ツール別の実績（p95）を蓄積。生成・大量処理・外部APIは「長時間候補」。
- **race パターン**: webhookは即200で受け、`waitUntil` 内で「タスク vs 約50秒タイマー」を競争：
  - 50秒以内に完了 → 結果を **Reply（無料）**。
  - 超過 → **「実行中タスクリスト」に登録**し、**「実行中です。完了後にメニューからご確認ください」と Reply（無料）**。本体は Queue で継続、完了で status を更新（**pushしない＝ユーザーがpullで確認**）。
- **実行中タスクリスト**（`running_tasks`: task_id/name/status/started_at/finished_at、利用者スコープ）はリッチメニューの postback → 無料Reply で表示。
- 文言例：「⏳『◯◯』を実行中です。数分かかる場合があります。完了したら下のメニュー【実行中タスク】からご確認ください（自動通知はしません）。」
- 真に長いバックグラウンドは `waitUntil` の上限を超えるため Cloudflare Queues の consumer で実行。

### 13-6. Push通数メータリング・無料枠残量・任意push

**通数メータリング（LINEメッセージ枠。AIコスト5-3とは別カウンタ）**
- OA単位で月間push通数をカウント（`channel.push()` 等の送信で加算）。プラン枠（200/5,000/30,000通）と比較し**残量**を算出。
- 枠到達時：コミュ/ライトは**配信不可** → pushをブロックしpull案内へフォールバック＋管理者へ警告。スタンダードは従量課金（〜¥3/通）で警告。
- 可視化：`tenantctl line-usage <id> --json`（送信/枠/残量）。ユーザーにもエージェントが提示（例「今月のpush 37/200通、残163通」）。

**ユーザー任意のpush（オプトイン）**
- 既定はpush最小（無料Reply中心）。ユーザーがエージェントに指示して**能動push購読**を作成（例：毎日1回の残タスク報告、日報）。実体は 13-3 のスケジュール＋`channel.push()`。
- これら定期pushは通数にカウント。残枠を尊重し、枠切れ時は pull案内/翌月送信へ切替。
- 登録：`push_subscriptions`(user_id/種別/周期/channel)。CLI 例 `tenantctl task add <id> --type daily-report --time 18:00`。

## 14. エージェント間連携（A2A・Discord）

会社をまたいでエージェント同士が連携する**オプション機能（上位プラン・相互同意制）**。
テナント分離・機密保持を崩さないため、**生のグループ会話で「立ち聞き」させない**。当社の非中継原則どおり、**A2Aの本文も当社を通さず、当社は同意検証と接続トークン発行・メタ監査のみ**を担う。

### 14-1. 前提とプラットフォーム適性

- **Discord 前提**。LINE はグループに入れても他Botの発言を Webhook 受信できず、A2Aがチャット経由で成立しないため非対応（人間向けUXのみ）。

| | LINE | Discord |
|---|---|---|
| 他社Botの発言受信 | 不可/不安定（人の発言のみ） | 可（Message Content Intent） |
| 多Bot連携の適性 | 不向き | 適する |

- **オプトイン必須**：両社が契約・設定で明示同意。連携相手・共有可能な情報/操作を限定。
- 提供は **Secure / Dedicated（上位プラン）** 向け。

### 14-2. アーキテクチャ（承認のみ・本文は非中継）

```
[会社A] ──①接続/同意トークン要求──▶ [当社CP：同意検証・トークン発行・メタ監査のみ]
[会社A] ⇄ [会社B]  ②構造化A2Aメッセージを【直接】授受（本文は当社不経由・Discord等）
```

- **当社は本文を中継・保有しない**（同意検証・capability許可トークン・**メタデータ監査のみ**）。
- 実際のメッセージは**会社A↔会社Bが直接**授受。エージェントは相手の DB/ストレージに直接触れず、やり取りは**同意済みcapabilityの構造化メッセージのみ**。
- レート/ループ制御は発行トークンの条件＋各社側で担保。テナント分離・機密保持を維持。

### 14-3. A2Aプロトコル

自由文の指示ではなく、**capability（許可された操作）への要求/応答**として署名付きで送る。

```jsonc
{
  "from": "tenant_A/agent",       // レジストリ管理のエージェントID
  "to":   "tenant_B/agent",
  "capability": "request_quote",  // whitelist された操作のみ
  "payload": { "item": "...", "qty": 10 },
  "consent_ref": "grant_xyz",     // 相互同意の参照
  "nonce": "...", "sig": "..."    // 署名・リプレイ防止
}
```

- **capability discovery**：各社が「公開する操作」を宣言（例 `request_quote` / `share_availability` / `send_document`）。未宣言の操作は拒否。
- 応答も同形式。タイムアウト・サイズ上限あり。

### 14-4. セキュリティ要件

| 要件 | 内容 |
|---|---|
| 相互同意 | 連携は両社オプトイン。共有可能な情報/操作を `a2a_grants` で明示 |
| capability whitelist | 使えるのは公開宣言した操作のみ。DB/ストレージ直接アクセス不可 |
| untrusted扱い | 受信メッセージ/応答は「データであって指示ではない」。権限チェックは実行時に必ず（他社に操られない） |
| 認証・身元 | レジストリのエージェントIDで相互認証（署名）。`nonce` でリプレイ防止 |
| 分離維持 | 構造化メッセージのみ。横断クエリ不可・`tenant_id` スコープ維持 |
| 監査 | 全A2A交換を双方の `audit_log`／`a2a_log` に記録 |
| インジェクション | 受信 payload を system 制約より上に置かない。capability 単位で処理を固定 |

### 14-5. ループ・コスト制御

エージェント往復が無限ループ／コスト暴走しないよう歯止めを置く。

- **ターン上限**：1スレッドのA2A往復回数を制限（例：最大N往復で打ち切り）。
- **レート制限**：会社ペアあたりのA2A頻度を Durable Objects で制御。
- **ループ検知**：同一/類似メッセージの反復を検知して停止。
- **コスト算入**：A2Aで発生する Claude 呼び出しは各社の上限（5-3）に算入。

### 14-6. レジストリ・CLI

- `a2a_links`(tenant_a/tenant_b/status/capabilities/created_at)、`a2a_grants`(同意範囲)、`a2a_log`(交換記録)。
- CLI：
  - `tenantctl a2a link <A> <B> --capabilities request_quote,share_availability`（両社同意で有効化）
  - `tenantctl a2a unlink <A> <B>`
  - `tenantctl a2a log <A> --json`

### 14-7. ユースケース・位置づけ

- 企業間の日程調整、見積/発注のやり取り、同意済み文書交換、団体間連携 等。
- **B2Bエージェントネットワーク**として差別化になり得るが、分離・機密と直結するため、**Discord前提＋制御A2A＋相互同意の専用オプション**として慎重にスコープを切る。

### 14-8. グループA2Aの参加者制限（承認アカウントのみ）

グループ（複数社）でのA2Aは、参加者を**承認済みアカウントのみ**に限定する。チャンネル在席ではなく**当社署名トークンの保有者でグループを定義**する。

- **参加資格**：当社レジストリ管理＋**ライセンス active＋整合性OK（17-9b）＋当社アクセスOK**の承認アカウントのみ。グループ定義（メンバー一覧）は当社コントロールプレーンで管理。
- **署名付きメンバーシップ・トークン**：参加・送受信には当社発行の**短命の署名トークン**が必要。発行条件＝上記の承認状態。各エージェントは受信メッセージの**メンバーシップ・トークンを相互検証**し、**トークンを持たない相手は無視/拒否**。
- **トランスポートと信頼境界の分離**：Discordチャンネルに第三者bot/アカウントが混入しても、**有効なトークンを出せないため承認エージェントは相手にしない**。A2Aの「グループ」は token 保有者で定義し、在席では定義しない。
- **allowlist**：各エージェントは承認済みピアID（公開鍵）の allowlist を当社から受領。非allowlistからのメッセージは破棄。
- **失効＝即時離脱**：suspended／ライセンス失効／改変／アクセス剥奪 → メンバーシップ・トークンの**発行停止＋失効リスト**。短命なので即座にグループから排除。
- **入会承認**：グループへの追加は**当社の承認＋既存メンバーの同意**が要る（14-4のオプトイン）。「誰が」＝メンバーシップ・トークン、「何を」＝capability、で二重に絞る。
- レジストリ：`a2a_groups`(group_id/members/created_at)、`a2a_membership`(group_id/tenant_id/status)。CLI：`tenantctl a2a group create <g>`／`group add <g> <tenant>`（承認チェック）／`group remove <g> <tenant>`／`group members <g> --json`。

## 15. 組織記憶（ナレッジ蓄積）と団体別人格

エージェントが**団体ごとの知識・特色・意向を自然に蓄積**し、**人が入れ替わっても引き継ぐ**ための仕組みと、**団体ごとの人格（プロンプト）調整**。

### 15-1. 組織記憶（Organizational Memory）

- **スコープ**: テナント（団体）の **company スコープ**（2-7）に保持。個人に紐づかないため、**メンバー交代・退任後も記憶が残り引き継がれる**。
- **蓄積するもの**: アップロード資料・議事録・成果物、会話から抽出した決定事項・FAQ・団体の方針/好み。
- **自然なストック**: 会話後に軽量モデルが「保存すべき事実か」を判定 → ナレッジへ追加。重要な決定は**人の確認つきで確定**（AIの自動確定はしない方針を維持）。
- **検索（RAG）**: 応答時にナレッジをベクトル検索（埋め込み＋reranker）して関連箇所を文脈に含める。原文の所在も提示。
- **団体プロファイル**: 蓄積から「方針・好み・重要事実」を定期的に要約し `tenant_profile` に保持。これを人格プロンプトに織り込む（15-2）。

実装：
- ベクトル化＝Workers AI 埋め込み（`bge-m3`/`plamo`）、再ランク＝`bge-reranker`。ベクトルストアは Cloudflare Vectorize（既定）／pgvector 等に差し替え可（ロックイン回避・12章）。
- レジストリ：`knowledge`(tenant_id/source/text/embedding/**scope**/created_at)、`tenant_profile`(要約/更新日時)。
- CLI：`tenantctl knowledge add <id> --file ...`、`tenantctl profile show <id>`。

### 15-2. 団体別の人格・プロンプト調整

- `buildSystemPrompt` がテナントの **persona 設定** と **団体プロファイル** を差し込み、「その団体らしい」応答にする。
- 設定項目（`agent_persona`／テナント単位）：エージェント名・トーン（敬語/フランク）・口調/言語スタイル・呼称・団体固有ルール・避けたい表現。
- 管理画面/CLIから編集（`tenantctl persona set <id> ...`）。**設定配信のみでデプロイ不要**。
- **ガードレールは不変**：可変なのは system prompt の「人格・スタイル部分」のみ。権限・スコープ・破壊的操作禁止・プロンプトインジェクション対策などの**安全制約は固定で上書き不可**。

### 15-3. スコープと位置づけ

- 個人の会話履歴＝**personal** スコープ、組織記憶＝**company** スコープ（2-7）で分離。前者は本人のみ、後者は団体で共有・継承。
- 組織記憶・人格調整は全プランで基本提供。蓄積容量・高度RAGは上位プランで拡張。

## 16. 技術ハードニング（技術リスクの完全カバー）

総点検で挙がった技術リスクを、設計として閉じる。各項目に具体的な実装方針を定める。
**PIIマスキング・データリージョンは任意選択**（プラン/テナント単位）。

### 16-1. PIIマスキング（任意・選択式）

- **オプトインで有効化**。AI（特に Claude＝越境）へ送る前に個人情報を検出・マスクし、必要時に復元する。
- 対象：氏名・連絡先・住所・子どもの学年等。検出は辞書＋パターン＋（任意で）軽量NER。
- **マスキング監査**：前後の長さ・件数をログ（本文は残さない）。
- 既定OFF（簡易・低コスト）／機微案件はON。`tenantctl masking set <id> --on`。
- 暗号化（8章）・組織記憶（15章）とも整合：マスク済みテキストを保存/送信。

### 16-2. データリージョン選択（任意・個別設定）

- dedicated/個別案件で**保存リージョンを選択**可能（例：日本国内）。
- 手段：Cloudflare の **Jurisdiction 制約**（D1/R2 のデータ所在地）・R2 ロケーションヒント・専用アカウントのリージョン設定。
- AI処理：Anthropic 等にリージョン/データ所在オプションがあれば指定。無い場合は「AI処理は国外」を**明示同意**化（16-1のマスキング併用で送信内容を制限）。
- レジストリ `region` 設定。`tenantctl region set <id> --jurisdiction jp`。
- 国内限定要件が厳しい社は **dedicated＋国内リージョン＋マスキングON** で対応。

### 16-3. テナント分離の保証（プール型D1の横断防止）

- 全クエリは `tenant_id` を必ず注入する**リポジトリ層のみ経由**（生SQL禁止）。
- **横断不可の自動テストをCIで強制**（他テナントIDでのアクセスが0件になることを検証）。
- 機微社は isolated/dedicated で物理分離へ昇格可。

### 16-4. プロンプトインジェクション多層防御

- 外部API応答・RAG結果・A2Aメッセージ・組織記憶は**「データ」境界**で囲い、system制約より上に置かない。
- 権限チェックは**実行時に必ず**（モデルの判断に依存しない）。破壊的操作はツール化不可。
- 出力フィルタ（機密の漏えいパターン検知）。
- 100%は不可と前提し、**被害の局所化**（権限最小・確認・監査）で受け止める。

### 16-5. 組織記憶の健全性

- **出所(provenance)** を全エントリに記録（誰が・いつ・どの会話/資料から）。
- 重要事実は**人の確認つきで確定**（自動確定しない）。
- **修正/忘却(forget)**：誤情報の訂正・削除。`tenantctl knowledge forget <id> <entry>`。
- 取り込み時にモデレーション（汚染・矛盾検知）。

### 16-6. 実行・無料枠制約への対応

- 会話履歴・上限カウンタは **D1**（KV無料枠の書込1,000/日制約を回避）。
- 長時間処理は **Cloudflare Queues** の consumer で完遂（`waitUntil` 上限を超えない）。webhookは**5秒以内に200**を返す。
- DOは Paid 前提機能として扱い、無料時は D1 ソフト上限で代替（2-6）。

### 16-7. バックアップ鍵管理・DR

- 暗号化退避の**鍵は別トラストドメイン**（KMS/別アカウント）に保管・バックアップ（鍵紛失＝復元不能を防止）。
- **リストア訓練を定期実施**（新規アカウントへ丸ごと復旧、10章）。`tenantctl restore --to` を検証。

### 16-8. 依存分散

- インフラ：ロックイン回避（12章・S3互換/SQLアダプタ）。
- AI：用途ルーティングで多プロバイダ切替可能（Anthropic障害時の代替）。
- チャネル：LINE ↔ Discord 切替。

### 16-9. 技術リスク対応マトリクス（総点検の閉じ込み）

### 16-10. 脅威モデル対策（追加防御）

詳細な攻撃分析は [04_threat-model.md](04_threat-model.md)。実装前に組み込む正式対策。

**最優先（致命的攻撃の防止）**
- **署名鍵（指摘#4・最重要）**：**Workers内で署名しない**設計へ。Ed25519秘密鍵は**外部KMS/クラウドHSM、またはオフライン署名専用プロセス**に置き、Workersは署名を依頼するだけ（公開鍵のみ配布）。リースは短命＝署名は低頻度なので外部署名のレイテンシも許容。**素のper-Worker Secretsは不可**（Worker編集者＝Secret編集者で職務分離不能）→ やむを得ずWorker内署名なら **Secrets Store＋プロダクトスコープ＋アクセス監視＋`kid`ローテ必須**。最小アクセス・`kid`即失効・**署名発行の異常監視**。CF外セカンダリは別`kid`。
- **管理者承認ゲート（徹底）**：**重要な変更は管理者（オーナー）の承認を必須**にする。`tenantctl` は「**提案→オーナー承認→実行**」の2段（**オーナー承認鍵で署名**）。
  - 対象（重要な変更）：config publish/promote、署名鍵の生成/ローテ/失効、break-glass発行、テナントstatusの手動上書き、破壊的操作（テナント削除・`restore --to`）、セカンダリ/DNS切替、管理権限の付与/剥奪。全件監査。
  - ※ソロ運用中はオーナーが提案・承認の両方を行う「**確認＋記録ゲート**」。チーム化で提案者≠承認者の職務分離へ移行。
- **段階配信の保護**：configリリースは**管理者承認＋署名＋Ring0テスト先行＋即ロールバック**（17-9）。
- **サプライチェーン**：client/CP は**依存ピン留め・SBOM・署名ビルド成果物・レビュー・最小依存**。

**IP/データ保護**
- **テナント別カナリア透かし**を config/prompt に埋め、流出元を特定→法的対応。
- **owner_id/scope はデータ層で強制**（プロンプト依存にしない＝インジェクションで越えられない）。
- **system prompt に機微IPを置かない・最小化**（吐かせ攻撃の被害限定）。

**コントロールプレーン**
- `/lease` は **bootstrap資格情報必須＋レート制限**。bootstrap資格情報の**ローテ＋異常検知**。
- config配信・リースは**署名検証必須**（MITM/改竄を無効化）。

**運用**
- **break-glass は管理者承認＋顧客可視の監査**。
- メタデータは**非信頼前提**＋スキーマ検証（本文混入防止）。

**段階導入の見直し（指摘#4：購入待ちにせず今すぐやれることが大半）**

開発機1台に**署名鍵＋管理者権限＋全顧客CFトークン**が集中＝壊滅的単一障害点（自動修復17-9bがあるぶん被害増幅）。以下は**メインPC固定を待たずに着手**する。

- **今すぐ（ハード固定不要）**：
  - 署名鍵を**CF外KMS/Secrets Store**へ（上記・素のWorker Secretsから外す）。
  - **全アカウント（管理者・CF・Anthropic・LINE）に FIDO2 ハードウェアキー（YubiKey等）でMFA強制**。※これは「PCの固定」ではなく既存PCで今日できる最も費用対効果の高い一手。
  - **鍵操作は別ユーザープロファイル＋そのプロファイルにのみ鍵アクセス**を与える（完全な専用端末でなくても大半のリスクが下がる）。
  - **時間差ゲート**（ソロでの職務分離の擬似実装）：鍵生成/ローテ・config publish・status手動上書き・`restore --to` は「**発行→N分後に別チャネル/別デバイスで確認して実行**」。1台侵害でも即時に全実行できない。チーム化後に提案者≠承認者へ。
- **本番まで後ろ倒し（PC確定後）**：完全な専用端末への固定。
- **本番ゲート（#4）**：①署名鍵がCF外KMS or Secrets Store ②全アカウントにFIDO2 ③鍵操作が専用プロファイル＋ハードウェアキー — **この3つが揃うまで本番提供を開始しない**（11章）。

**構造的に完全には防げない2点（被害は1顧客に限定）**
- (1) 顧客によるclient改造での無料/frozen継続、(2) 配信configの抽出。→ 難読化・カナリア透かし・**価値を更新/組織記憶/ネットワーク/サポートに置く**・法的対応で緩和。致命傷は鍵/管理者奪取側であり、そちらを最優先で守る。

### 16-11. ループ・暴走コスト防止（多層バウンド）

**原則：すべての反復/再帰/定期/リトライ経路に"必ず止まる"ハード上限を置く（fail-safe to STOP）。** 暴走コストの主着地点は顧客の Anthropic(BYOK)/LINE/CF と当社CP。月次cap（5-3）は粗いので、**単一ターンを止める細かいガードを併用**する。悪意の有無に関わらずバグでも暴走し得る前提で設計する。

| ループ/暴走源 | 負担者 | ガード |
|---|---|---|
| エージェントのツール連鎖が止まらない | 顧客(Claude) | **1ターンのツール実行回数上限**（例8）＋強制終了（02 `MAX_STEPS`） |
| 1ターンで大量/並列にClaude呼出 | 顧客(Claude) | **per-turnコスト予算**で打切り＋**原子カウンタ(DO)で月次capを正確に**（並列overshoot防止） |
| 定期タスク(cron)が高頻度/自己再スケジュール | 顧客 | **最小実行間隔＋1日あたり実行回数上限＋per-taskコスト予算** |
| 外部コネクタ/マクロの再帰・自己呼出 | 顧客 | **呼出深さ上限・自エンドポイント呼出禁止・コネクタのレート制限** |
| Webhook再送で重複実行 | 顧客 | **冪等キーで重複排除**・先にACK（waitUntil） |
| AI/外部エラーのリトライ暴発 | 顧客 | **リトライ回数上限＋指数バックオフ＋サーキットブレーカ** |
| 自己メッセージ→自Webhook→再実行 | 顧客 | **Bot自身/自テナント発イベントを処理しない・ループ検知** |
| A2A往復ループ | 双方 | ターン上限＋ループ検知（14-5） |
| `/lease`・config配信の連打 | **当社CP** | **テナント単位レート制限**＋client必須キャッシュ（無料枠死守） |
| 監視triageの再発火 | 当社 | アラートのdebounce/クールダウン・dedupe |
| 段階配信のflap（自動ロールバック⇄昇格） | 当社 | クールダウン＋最大試行＋停止して人へ |
| 自動増枠の暴走 | 顧客 | **auto-topupを持たない**（増枠は管理者承認の手動のみ・5-3） |

- **多層で必ず止まる**：ステップ上限 → per-turn予算 → レート制限 → 月次cap → サーキットブレーカ。
- 暴走検知時は**メタデータ（9-0）でアラート→自動throttle/停止＋顧客通知**。
- 攻撃面としても整理（04の⑤コスト爆撃と連動）。

## 17. 運用モデル・ライセンス制御・IP保護・オンボーディング

ベース運用形態の中核。**当社は承認（ライセンス）とロジック配信のみ**、**顧客データ（本文）は非中継**、**Anthropic/LINE は顧客が直接やり取り**。これで権限集中・ゼロ保持・レイテンシ・各社ToS・キルスイッチの法的根拠を同時に最適化する。

### 17-1. 役割分担

| | 顧客 | 当社 |
|---|---|---|
| アカウント所有（CF/Anthropic/LINE） | ◯ | ―（作成支援は17-6） |
| Anthropic/LINE との通信 | **◯（直接）** | ×（キーも本文も持たない） |
| データ保有（at rest）・処理 | ◯（自社アカウント内で実行） | **×（非中継・非保有）** |
| 中核ロジック（IP） | 実行のみ（配信を受領） | ◯（ライセンスゲートで配信） |
| 承認（ライセンス） | × | ◯（発行/停止） |
| Anthropic/LINE のアカウント管理 | ◯（**完全に自己管理**） | **×（権限を一切持たない）** |
| Cloudflare のアクセス | 必要時のみ**時限JIT付与→失効**（常設ゼロ） | 初回構築・復旧時のみ作業（17-5） |

> **委任権限を受けるのは Cloudflare だけ**。Anthropic・LINE は顧客が完全に自己管理し、当社はアクセス権も本文も持たない。これにより「第三者による代理管理・データ取扱い」のToS論点を **Cloudflare 一本に限定**できる（代理店/MSP モデルとして最も通しやすい領域）。

### 17-2. 配信モデル（事前配備コード ＋ エフェメラル config）

> **訂正（第三者指摘#2を反映・要PoC検証）**：Workers ランタイムは `eval`/`new Function` 等の**動的コード実行を禁止**するため、「**コードをエフェメラル配信してメモリ上で実行（常駐保存しない）**」は**技術的に不可能**。コードは**事前配備が前提**。エフェメラルに配信するのは**設定（データ）のみ**。（Workers for Platforms は動的実行可だが、実行・課金・入口が当社側に戻り非中継/COGS≒0/越境を破壊するため不採用。）

- **構成（B＋A）**：顧客アカウントに**固定のエージェント/インタープリタWorker（難読化）を事前配備**（CF API経由・17-9bのreconcileと同一）。`/lease` で配信するのは**署名済み config（プロンプト・ツール定義・人格・パラメータ＝データ）のみ**で、Worker がそれを解釈して挙動を変える。**コードは配信しない**。
- 非中継・COGS≒0・データ非保有・**ライセンスによる機能停止は全て維持**（これらはコードのエフェメラル性に依存しない）。Anthropic/LINE は顧客が直接、本文は当社を通らない。
- **IP保護の定義を変更**：「コードを見せない」→「**見られても価値が承認ゲート・更新・蓄積に依存する**」。
  - 事前配備コードは顧客が自アカウント（CF API/管理画面）で**読める**。難読化は手間を上げるだけ。config も顧客が読める（≒プロンプト）。
  - したがって**モートは静的コード秘匿ではなく ①承認ゲート（ライセンス） ②継続アップデート ③組織記憶 ④ネットワーク**。config には**テナント別カナリア透かし**で流出元を追跡。
- **PoCで最初の48hに検証**：①固定インタープリタWorkerを wrangler 配備し**署名済み config(JSON) だけで挙動が変わる**ことを実証（コード配信ゼロ） ②**ライセンス無効→config配信停止→機能停止／データ無停止**（17-3）が成立するか ③canary で**顧客が config をどこまで読めるか（IP露出の実測）**→「Aで十分か／Bの難読化の効き」を判断。

### 17-3. ライセンスによるキルスイッチ（機能のみ停止・データは人質にしない）

```
エージェントWorker ──①ライセンス/設定 要求──▶ 承認サーバ(当社・多リージョン)
                  ◀─②署名リース＋エフェメラル設定（本文なし）──  ※サブスク状態を判定
   有効 → エージェント稼働 ／ 無効・未発行 → エージェント機能のみ停止
```

- **契約無効・未払い・規約違反 → リース/設定の発行停止 → エージェント機能のみ停止**。**顧客のアカウント・データ・他システムは無停止**（人質化しない）。
- 段階制：**警告（猶予リース）→ 機能停止**。是正/入金で**即時復帰**。
- **暗号鍵による強ロックは採用しない**（顧客の自社データへのアクセスを奪わない＝優越的地位の濫用・自力救済の懸念を回避。18章）。
- **grace＝更新日（月額制に整合）**：リースに **`paid_through`（＝次回更新日）** を載せ、**クライアントは `paid_through` まで稼働**（当社CPの到達可否に依らない）。
  - **到達できる時**：短命JWT（例1h）を頻繁にリフレッシュ＝**失効/解約は約1hで反映**（セキュリティ）。
  - **到達できない時**：直近リースの **`paid_through` を信頼して更新日まで稼働**（当社の障害でも当月は止まらない）。
  - **更新日に**：支払い確認 → 次の `paid_through` で継続／未払い → 更新日で停止（決済ラグ用の小さな猶予のみ）。**支払い確認でき次第、自動再開**。
- **fail-open ルール**：クライアントは「**明示的 suspended（署名付きNo）→ 即ロック**」「**到達不能 → `paid_through` まで稼働継続**」を区別。**当社/CFの障害がロックの引き金にならない**。全CF障害時はエージェント自体も止まるが、復旧時に**キャッシュ済みリースでシームレス再開**。
- **自走クライアント（支払い中は当社不在でも稼働）**：クライアントは**最後の正常リース＋設定を永続キャッシュ**し、**当社に接続できなくても `paid_through` まで走り続ける**。当社CPは**「更新」と「enforcement」専用**でランタイムには不要。
  - 当社CFアカウント事由でも、**支払い中の顧客は更新日をまたいで継続**できるよう、更新は**CF外セカンダリが肩代わり**（17-4）。**完全オフライン恒久は採らない**（未払いは更新日で必ず止まる）。

### 17-4. 多リージョン・アクティブ-アクティブ（全社SPOFの回避）

- 承認/配信サーバは **Cloudflare Workers** で実装＝**既定で世界中のエッジに分散実行（計算は最初からマルチリージョン）**。鍵は Workers Secrets（グローバル）、サブスク状態は **KV（グローバル複製・各地でローカル読取）**。
  - **承認は「ステートレス署名＋グローバルKV読取」**にすることで、**ホットパスは初日から全リージョン耐性**。「後でリージョン追加」は実質ゼロ工数。
  - **後付けで足すのは書込み/制御側（D1プライマリ）の冗長化のみ**（読取ホットパスはKVで既にグローバル）。初期はシングルでよい。
- **CF外・セカンダリ管理（長期resilience要件）**：**当社CFアカウント事由（停止/解約/ban）でも顧客が止まらない**ことを恒久方針とするため、CF外のサーバー（別クラウド/VPS・別法人/別契約）に**縮小版の承認・設定配信**を置き、クライアントの**セカンダリ・エンドポイント**にする。
  - 役割：自走クライアント（17-3）が**「更新」と「enforcement」を継続**するための経路。プライマリ（CF）が長期到達不能でも、セカンダリで版更新・課金反映を続けられる。
  - 効くのは「**長期CP障害／当社CFアカウント事由**で顧客CFは生存」ケース（全CF障害はエージェント自体が止まるため別問題＝顧客側CF復旧待ち）。
  - 鍵は**別 `kid`** を持たせ独立失効可能に（鍵管理のトレードオフ）。CLI管理。
  - **復旧/移行フロー（別CFアカウント切替）**：当社CF障害中はセカンダリが承認/更新を肩代わり → **別CFアカウントに新プライマリを構築** → **安定ドメイン（例 `cp.example.com`）のDNSを新アカウントのWorkerへ向け替え** → クライアントは通常運用へ復帰、セカンダリは待機へ。
  - 前提：**制御プレーンのドメイン/DNSを当社CFアカウントから独立管理**（別アカウントへ付け替え可能に）。
  - 段階：**初期は CF単体＋自走クライアント＋`paid_through` で当月無停止を担保**。**当社アカウント事由で更新日をまたぐ継続が要るためセカンダリを用意**（鍵 `kid`・セカンダリURL・独立DNS を最初から想定）。
- **クライアント側に有効リースをキャッシュ**（例：1時間）。承認は「メッセージ毎」でなく「リフレッシュ毎」になり、瞬断も無感に吸収・コストも僅少。
- **可用性SLA**を定義し、停止時の**賠償（サービスクレジット）上限**を契約に明記。
- **Cloudflare 全体障害**は当社の制御外として**契約で免責**。あわせて**ローカルサーバーへのバックアップ（10-4）を推奨**し、最悪時も顧客手元で復旧できる経路を残す。
- 設定/ロジックの一斉配信は「正しさのSPOF」になり得るため、**段階配信（リング）＋テスト先行＋即時ロールバック**を必須とする（**17-9**）。

### 17-5. アクセスモデル（常設ゼロ＋時限JIT）

- **当社は普段アクセス権を持たない（常設ゼロ）**。CFへの権限は**初回構築・復旧など必要時だけ、顧客が時限・スコープ付きで付与 → 作業後に失効**（Just-in-Time）。
  - 理由：MSP/代理店パターンの核心は操作頻度でなく「**一事業者が多数の他者口座に常設特権を保持する構造**」そのもの。常設特権を撤廃すると、CFから見て"フリート運用するMSP"→"必要時に同意を得て支援するベンダー"へカテゴリが変わる。**鍵/トークン集中の単一障害点（集約攻撃面）も縮小**（04）。
- **初回構築**：JIT で一度だけ付与 → **即失効**（その後は鍵を持たない）。
- **鶏卵問題の回避**：復旧場面はエージェントが壊れていることが多く「壊れたエージェント経由で権限を渡す」が不成立 → **顧客がCFダッシュボードで手動トークン発行する経路を必ず残す**。
- **日常（課金ゲート・監視・config更新・自己修復）は当社アクセス不要で完全自動**：クライアントが**自口座スコープの自己更新トークン**（顧客口座内・当社非保持・当該Workerのみ編集可）でpull更新・自己再デプロイし、整合性は**client push**で報告（17-9b）。
- **Anthropic・LINE は顧客が完全に自己管理**（当社は権限なし）。本文は当社を通らない。

### 17-6. オンボーディング（顧客自己申込＋丁寧なガイド）

- **既定＝顧客自身がアカウント申込**（規約準拠）。**文書＋動画のステップバイステップ・マニュアル**で誰でも作成できるようにする。
  - 理由：Cloudflare 2.2.1(a) が「第三者に代わっての申込」を制限するため、**当社が代理で"申し込む"形は避ける**（18-1）。
- 難しい場合は**オンラインで同席し、画面共有で本人の操作を補助**（申込・同意は顧客本人が行う）。
- 完全な作成代行が必要なら、Cloudflare の**書面許可/パートナー契約を取得した上で**のみ実施。
- 作成後、`tenantctl onboard <id>` が冪等に：**JIT で一度だけ付与されたトークン**でアカウント接続→エージェントWorker配備（＋**自口座スコープの自己更新トークンを顧客Secretsに設定**）→ライセンス紐付け→疎通確認 → **付与トークンを即失効**（以降、当社は常設アクセスを持たない・17-5）。

### 17-6b. エージェントによるオンボーディング・ウィザード（操作をコピペまで削減）

顧客の手間を「アカウント作成＋トークンのコピペ」まで削るための、**ガイド付きウィザード**。

> **明記：各アカウントの作成は顧客自身が行う（代理作成不可）**。**Cloudflare ／ Claude API ／ LINE（または Discord）のいずれも、各社規約により顧客本人がサインアップ・規約同意・課金設定**を行う必要があり、当社・エージェントが代理で作成することはできない（第三者作成APIも存在しない）。当社は**作成手順の案内・入力の検証・自動配線**を担い、顧客の操作はコピペ中心に最小化する。

**フロー**
1. **顧客が Cloudflare アカウントのみ自己作成** → 当社を**メンバー/スコープトークンで管理に追加**（CF ToS準拠・17-6）。**顧客のCF側の手作業はこの「アクセス付与の1アクション」だけ**（メール招待 or トークン貼付。最小権限はスコープトークン推奨）。**課金不要＝無料枠で完結**（Workers Paid は拡大時のみ）。
2. 当社が**フル自動セットアップ**（CF API経由）：workers.dev サブドメイン登録／エージェントWorker デプロイ／**Workers AI バインド**（追加アカウント不要・無料枠・即利用可）／D1・KV・R2 作成・バインド／Cron Trigger／bootstrap Secret 投入／疎通確認。`tenantctl onboard <id> --wizard` 一発。以降、顧客はCFを触らなくてよい。
3. **Workers AI（無料）で"オンボーディング・エージェント"を起動** → これが残りを案内（**Claudeがまだ無くても進行できるブートストラップ**）。
4. オンボーディング・エージェントが**Claude API／LINE or Discord の作成を会話で誘導**（リンク・クリック手順・動画・スクショ）。可能な所は**ディープリンク/テンプレ**で短縮（例：Discordはbot招待OAuth URLでほぼワンクリック、LINEは手順案内＋自動webhook登録）。
5. 顧客は**取得したキー/チャネル情報をコピペ**（エージェント or 安全フォームへ） → 当社が**検証**（疎通テスト：Anthropic最小呼出／LINE `GET /v2/bot/info`／Discord token確認＝baku-pta のプレフライト相当） → 合格後**顧客の Worker Secrets へ自動投入＋Webhook URL等を自動配線** → **稼働開始**。
6. 完了後、エージェントは Workers AI から **Claude（顧客のBYOKキー）へ切替**。

**セキュリティ・規約**
- 入力されたキー/シークレットは**顧客自身の Worker Secrets に入る**（当社は保持・ログしない・末尾マスク）。「**自分の環境に入れる／当社は預からない**」と明示してフィッシング誤認を防ぐ。
- 全アカウントは**顧客自己作成**（代理作成しない）＝規約準拠。自動配線（Workers設定/Secret投入/Webhook登録）は**JIT付与での一度きりの構築（常設アクセスなし・終了後失効）**（規模時は#3のパートナー枠で正規化・18-1）。
- CLI：`tenantctl onboard <id> --wizard`（プレフライト検証→自動配線）。

### 17-7. コスト・可用性・依存の方針

- **データは顧客側**（DB/ストレージ・組織記憶/RAG・ベクトル）。当社は本文を**非中継・非保有**。
- **当社インフラ費は極小**：承認・設定配信は軽量（リクエスト/署名のみ）。中継しないため転送も発生せず、規模が出ても誤差。
- **依存（lock-in）の方針**：機能はライセンス依存で担保しつつ、**データは常に顧客の手元**。顧客保護として **SLA・データ可搬性（S3互換/SQL）・契約終了時の設定/データ引渡し**を明文化（公正性で優越的地位の濫用を回避・18章）。
- **当社CPコストの事前アラート**：正常利用でも課金が出そうな時は事前に把握する。**無料枠（Workers req/KV/Analytics 等）の70/90%でアラート**＋**成長予測で「Paid化が要る時期」を事前通知**（Cloudflare Billing/Usage通知＋GraphQL Analytics を cron 取得）。発生前に最適化（キャッシュ/サンプリング）か計画的な $5 昇格を選ぶ。`tenantctl cost-alert`。

### 17-8. 鍵管理（紛失によるアクセス不能の防止）

- **ライセンス署名鍵**（承認の根幹）：クラウドKMS＋オフラインの**ローカル冗長保管**、世代管理・ローテーション。**紛失＝全顧客がリース取得不能**になるため最重要。
- 顧客側の暗号鍵（暗号化オプション時）：**顧客がクラウド＋ローカルで冗長保管**。当社は保持しない。
- 事業譲渡時の鍵引渡し/ローテーション手順を文書化（19章）。

### 17-9b. 完全性・自己修復（client push＋顧客起点の巻き戻し・常設アクセスなし）

クライアントは攻撃者制御環境で動くため**完全な改ざん"防止"は不可**。検知は **client push（当社アクセス不要）**、修復は **顧客起点の巻き戻し（既定）／自動修復は opt-in**、権威確認は **JITのみ**。enforcement は自己修復に依存せず**リース側**（exp＋paid_through＋署名）が担う。

**(1) 継続シグナル＝client push（当社は顧客アカウントを触らない）**
- 顧客Worker が小さな署名付きレポートを送信：`agent_build`（埋込ビルドID）／`config_version`／`lease_jti`／`attest`（リースnonceに埋込鍵でHMAC＝anti-spoof）／`health`（数値メタのみ・本文/PIIなし）／bootstrap署名＋nonce。
- 用途：**申告のズレ／heartbeat途絶**を検知。詐称可能ゆえ"安価・継続・低干渉"＝誤操作・事故・detachに十分。
- **頻度（既定）**：1時間ごと（cron）＋アクティブ時はリース更新に相乗り。**detached＝連続2〜3欠落でflag**。コスト最優先は6〜12時間、重要顧客は15〜30分に可変。`last_seen` は D1/Analytics。

**(2) 修復＝顧客起点の巻き戻し（既定）／自動修復は opt-in**
- **既定：顧客が誤変更を正規版に戻せる**（ワンタップ「正規版に戻す」）。「**あなたの口座・あなたが制御・当社は署名済みの正規復元ポイントを提供するだけ**」＝データ主権の訴求が実態と一致。
- **opt-in：自動修復**（"壊れても勝手に直る"managed体験を望む顧客向け）。drift検知でクライアントが自動で正規版へ。
- どちらも**戻し先は当社署名済みの正規版（`config_version`＋`build`）のみ**。**自己再デプロイは"自口座スコープの自己更新トークン"（顧客口座内・当社非保持・当該Workerのみ編集可）＋Ed25519署名検証必須**（汚染config/注入で任意コードが乗らない）。**ライセンス検証はインタープリタ内に固定**＝巻き戻し/改変で無効化できない。
- データ復旧は**別経路（バックアップ・10章）**。当社管理の正規版はコード/バインド/Secret配線を復元、顧客の鍵は再入力ガイド。
- CLI/UI：`tenantctl heal <id>`（JIT/顧客承認時）／顧客向け「正規版に戻す」操作。

**(3) 権威確認＝JIT/オンデマンドのみ（常設アクセスなし）**
- 当社は**常設アクセスを持たない**（17-5）。CF APIでの実物読み戻し（Cloudflareの真実）は**疑義時/復旧時にJIT付与を受けた時だけ**実行（routineではない）。routineの整合は client push と（opt-in時の）自己修復が担う。

**enforcement（自己修復に依存しない）**
- 短命 `exp`：到達可なら失効/解約を即反映。`paid_through`：到達不可でも更新日まで稼働（当社/CF障害でも当月は止まらない）。署名：偽造不可。
- **未払いは更新日で必ず停止／client改造は被害が1顧客に限定**（04・脅威モデルで既決）。＝①を opt-in 化しても enforcement は無傷。

- **限界（正直）**：完全 detach した client は `paid_through` まで走る（有界・更新は止まる）。config を別Workerにコピーする「IP抽出」は対象外（透かし＋法的・04）。
- **最悪ケースの許容**：顧客が自社口座で改変を続け自動復旧を拒む場合は**法的措置**。被害は**その顧客の自アカウント内に限定**（データは顧客自身・テナント分離・A2A同意トークンの停止・コネクタegress制御で**第三者への外部被害なし**）ため、外部被害が無い限り許容。

### 17-9. 段階配信（リング）＋クライアント発pull更新【必須】

設定/ロジックは**バージョン管理**し、各テナントに**リング**を割当てて段階展開する。承認サーバはテナントの `ring` ＋「どの版がどのリングに出ているか（rollout state）」を見て、**該当版の設定を返す**だけ。

**更新はクライアント発pull（当社からのpushを廃止＝代理運用度を下げる）**
- **config（プロンプト/ツール/人格/パラメータ＝データ）**：クライアントが版差分を検知して**自動pull＋差し替え**（完全ゼロタッチ・当社アクセス不要）。
- **コード**：Workers は eval 不可のため pull だけでは差し替え不可 → クライアントが**自口座スコープの自己更新トークン**で**自己再デプロイ＋Ed25519署名検証必須**（当社署名済みアーティファクトのみ適用）。当社からのpush・常設アクセスは不要。コード変更は極力 config に寄せて低頻度化（稀に JIT 承認）。
- Ring 0（身内テスト）先行は維持。クライアントは**自分のringの版**をpullする。

| リング | 対象 | 役割 |
|---|---|---|
| **Ring 0** | **身内テストアカウント（必須）** `is_test=true` | 新版を**最初に受け取り本番前に先行検証** |
| Ring 1 | カナリア（少数・希望者） | 早期検証 |
| Ring 2 | 段階（10%→50%…） | 漸進展開 |
| Ring 3 | 全顧客 | 全面展開 |

- **テスト先行（身内）を必須化**：自社テストテナントを Ring 0 に置き、**必ず身内で確認してから**上のリングへ昇格。
- **ロールバック**：`config rollback` でリングを前版へ戻す→次のリフレッシュで正常版に。クライアントは**直前の正常版をキャッシュ**しフォールバック。
- **健全性ゲート**（任意）：リングのエラー率/health（メタデータ・9章）が悪化したら**自動でロールバック/昇格停止**。
- CLI：`tenantctl config publish v42`／`config promote v42 --ring 2`／`config rollback`／`tenant set <id> --ring 0`。
- 実装はWorkers+KVのロジックのみ＝**追加インフラ費ほぼ$0**。リスク低減が目的。

### 17-10. 実装スキーマ（リースJWT・KV・rollout state）

**リース JWT**（本文・個人データは含めない）
```jsonc
// Header
{ "alg": "EdDSA", "kid": "lic-2026-06" }      // Ed25519。kidで鍵ローテ／CF外緊急ホストは別kid
// Payload
{
  "iss": "cp.example.com", "sub": "tenant_abc", "aud": "agent-client",
  "iat": 1750000000, "nbf": 1750000000, "exp": 1750003600,  // 短命(例1h)→失効は約1hで伝播
  "jti": "ulid...",                                          // リプレイ防止・監査
  "lic": {
    "status": "active",            // active | grace | suspended
    "plan": "secure",
    "features": ["external_connector","broadcast"],
    "caps_ref": "caps:secure",     // 上限は参照（JWT肥大化回避）
    "ring": 0,                     // 段階配信リング
    "config_version": "v42",       // 発行時に ring×rollout で解決
    "paid_through": 1751328000,    // 次回更新日。到達不能時はこの日まで稼働
    "grace_sec": 86400             // 更新日後の決済ラグ用の小猶予（例1日）
  }
}
```
- クライアントの稼働判定：**到達可→短命`exp`でリフレッシュ（失効を速く反映）／到達不可→`paid_through`(+小grace)まで稼働**。`suspended` は即ロック。
- 検証：クライアントが**公開鍵（kid固定）**で署名検証＋`exp/nbf`＋`status`適用。秘密鍵は Workers Secrets `LICENSE_SIGNING_KEY_<kid>`。

**KV / Secrets / R2 キー設計**
```
KV  TENANT_STATUS  tenant:<id>     → { status, plan, features, caps, ring, updated_at }
KV  ROLLOUT        rollout:current → (下記 rollout state)
KV  CONFIG_INDEX   config:<ver>    → { r2_key, sha256, sig }
KV  REVOKED        tenant:<id>     → 1            // exp前の即時ハード失効(任意)
Secrets            LICENSE_SIGNING_KEY_<kid>      // 公開鍵は配布
R2                 config/<ver>.js                // 設定/ロジック実体・署名付き
```
- 状態読取は **KV（グローバル複製・各地ローカル）**、書込は低頻度。設定実体は **R2**（版別・ハッシュ/署名検証）。

**rollout state**（KV `rollout:current`）
```jsonc
{
  "channel": "default",
  "versions": {
    "v42": { "artifact": "config/v42.js", "sha256": "…", "published_at": 1750000000 },
    "v41": { "artifact": "config/v41.js", "sha256": "…", "published_at": 1749900000 }
  },
  "rings": { "0": "v42", "1": "v42", "2": "v41", "3": "v41" }, // 各リングが使う版
  "rollback_to": "v41",
  "health": { "v42": { "error_rate": 0.004, "samples": 1200 } }, // 9-0のメタから
  "updated_at": 1750000500, "updated_by": "tenantctl"
}
```

**発行フロー**
```
client → POST /lease { tenant_id, nonce }（顧客Worker内のブートストラップ資格情報で認証）
  CP: KV tenant:<id> と rollout:current を読む → version = rings[tenant.ring]
      active/grace → JWT署名(exp=now+1h, config_version, grace_sec)
      suspended    → { status:"suspended" }（明示拒否＝即ロック）
client: リースをキャッシュ → R2から config_version 取得(ハッシュ検証) → ローカル実行(Anthropic/LINE直接)
        exp前にリフレッシュ／到達不能は grace 内で継続（17-3 fail-open・自走）
```

### 17-11. 運用自動化（導入時を除き人手なし）

承認・課金・監視の**ルーティンは全自動**。人手は「設定リリース判断」と「インシデント例外」のみ。

- **課金→enforcement の自動ループ**：決済プロバイダの **webhook → `TENANT_STATUS` の status 更新 → リース発行/停止**。人手承認なし。
- **dunning（督促）自動化**：請求失敗 → **自動で警告通知（猶予リース）→ 段階督促 → 猶予超過で機能停止**。誤停止防止に**リトライ・通知・grace** を挟む。
- **監視**：決定論ガード＋Workers AI 巡回は常時自動、Claude は異常時のみ。自動対処（throttle/quarantine/suspend）後に**例外だけ人へエスカレーション**。
- **段階配信の健全性ゲート**：リングのメタ（9-0）が悪化したら**自動で昇格停止/ロールバック**。版の昇格"開始"判断のみ人手（＝製品リリース、ルーティンではない）。
- **無料枠厳守の実装規約**：リースは**クライアントキャッシュ**（メッセージ毎に承認を呼ばない）／メタは**サンプリング/集約**／カウンタは**D1**（KV高頻度書込を避ける）。これで usage overage を出さない（固定費は拡大時の Workers Paid $5 のみ）。
- すべて **CLI（`tenantctl`）** から設定・確認可能。

## 18. 法的ポジション（最終・顧問弁護士確認予定）

本サービスは「**データの操作・管理は顧客が行い、それを便利にするシステムの提供・保守を当社が担う**」事業として位置づける。下記方針は**完成後に顧問弁護士のレビューを経て確定**する（条項文言・届出要否を含む）。

| 論点 | ポジション・対応 |
|---|---|
| **個人情報の越境移転・委託** | **Anthropic/LINE は顧客が直接契約・通信**し、**本文は当社を通らない（非中継）** → 当社は本文の**処理委託先にならず、ソフトウェアのライセンサー＋運用支援者**の立場。越境同意は**顧客↔各社の契約**で整合（BYOK）。当社の支援アクセスに伴う限定的処理は委託契約(DPA)でカバー |
| **キルスイッチ・優越的地位の濫用/独禁法** | 停止対象は**エージェント機能（ライセンス）のみ**。**顧客のアカウント・データ・他システムは停止しない**（人質化しない）。停止権・通知を契約に明記、データ可搬性を保証し公正性を担保。**暗号鍵による強ロックは不採用** |

| **プロバイダ責任制限法・コンテンツ責任** | データの操作・管理・コンテンツ責任は**顧客**（自社アカウント）。当社は**システム提供・保守**の立場。違法コンテンツは通知対応(takedown)・必要に応じ通報 |

| **各社 利用ポリシー準拠** | アカウント管理とは別に、**当社が配信するエージェント挙動が各社ポリシーに準拠**（Anthropic Usage Policy／LINE 配信ルール）。ライセンス配信ロジック＋利用規約で担保 |
| **役割の明確化** | 顧客＝管理者・データ保有者・運用主体／当社＝**ライセンス提供・導入支援・共同保守**。委任範囲を RACI で契約化 |

> いずれも当社の現方針であり、**最終確定は顧問弁護士の確認後**とする。

## 19. 事業譲渡可能性（ホスト機能の売却前提）

ホスト機能（承認・ロジック配信＋ライセンス＋レジストリ＋顧客契約）を**資産として売却できるよう**、譲渡可能性を設計に織り込む。

| 譲渡対象 | 方式 |
|---|---|
| **Cloudflare アカウント** | 新所有者を **Super Administrator に追加 → 旧所有者を削除**で実質オーナー移転。ドメイン/ゾーンも別アカウントへ移管可。Enterprise は正式な所有権移転手続あり |
| **Anthropic/AI アカウント** | 法人ごと譲渡 or 再契約・再キー化 |
| **コード/IP・ライセンス署名鍵** | 非公開リポジトリと**署名鍵**を引渡し（or 買主鍵へローテーション）。**買主がリース発行を継続できる**ことを保証（手順書化） |
| **顧客契約** | 契約に**譲渡条項**を入れ、M&A 時に承継可能に |
| **顧客データ** | **顧客アカウント内に残る＝売却対象外**。買主はデータ責任を引き継がない |

- **推奨**：ホスト資産を**法人（事業会社）に集約**し、株式譲渡/事業譲渡で一括移転（アカウント単位の移管より確実）。
- **個人情報**：レジストリ/監査ログ内の管理者連絡先等は、**事業承継に伴う提供（個人情報保護法27条5項の例外）**として承継可能（要顧問弁護士確認）。
- **データ非保有が売却を容易にする**：顧客データを抱えないため、データ移転の同意問題・債務リスクが小さく、買主の評価（バリュエーション）が上がる。アカウント管理者の変更（譲渡）も、データを伴わないぶん手続が軽い。

### 19-0. サービス終了（ウィンドダウン）も可能

当社がライセンス発行を握るため、**事業/エージェント機能を予告＋十分な猶予をもって秩序立てて停止**できる（売却と並ぶ出口）。

- **手順**：終了告知（契約の**任意解約条項＋通知期間**）→ 移行期間は通常稼働 → **サンセット日にライセンス発行を停止** → 各顧客は更新日/猶予で機能が順次停止。
- **顧客は取り残されない**：**データ・アカウントは顧客保有**のため、停止してもエクスポート/移行が可能（失うのは「エージェント機能」のみ）。
- **配慮オプション**：サンセット時に**移行用の長期/特別ライセンスを一括発行**して猶予延長、または引渡し支援。
- **未払いロックの仕組み＝事業全体のOFFスイッチ**でもある（対称性）。
