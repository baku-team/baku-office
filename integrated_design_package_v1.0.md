## 1. 概要とアーキテクチャ

### 1.1 サービス概要

baku-office は **「ポータブルコア＋パーツ」アプリケーション基盤**である。コア（DB・ストレージ・AI・エージェント・認証）を Port で抽象化し、業務機能は Part（パーツ）として後から載せ替える。コアは環境も業務も知らないため、**同一コードでフルクラウド〜完全オフライン（ローカルLLM）まで**動かせ、団体・用途ごとにパーツと UI を差し替えて配布できる（§1.4）。

**現在の標準同梱パーツは「会計・庶務（非営利団体向け／LINEエージェント連携）」だが、これは一例**である（在庫・予約・問合せ対応 等、別の業務パーツへ載せ替え可能）。本書の §2 以降は、この標準パーツ群を Cloudflare 上で自己ホスト配布するときの仕様を述べる。

運用は自己ホスト型：**業務データ（会計記録・名簿・ファイル等）はクライアント団体自身のCloudflare内に保管され、当社（ホスト）はそのデータに一切アクセスしない。** 当社が提供するのはソフトウェア（難読化バンドル）と動作のためのライセンス、課金・サポートである。

### 1.2 ホスト／クライアントの関係と信頼境界

- **当社（ホスト）**：ソフトウェアライセンサー兼配布元。非公開ソースの管理、難読化バンドルの公開配布（§3）、ライセンス発行（§4）、バージョン・通知の配信（§13.1）、B2B顧客情報（団体名・代表者・連絡先・請求情報）の管理。クライアントのCloudflareへのトークンも、データへの到達経路も持たない。
- **クライアント（団体）**：個人情報取扱事業者。自身のCloudflare／GitHubアカウントを保有し、アプリを自分で配備、業務データを自分で保持・管理・バックアップ。

境界を越えるのは「配布物・ライセンス・バージョン／エンタイトルメント情報・通知」（当社→クライアント）と「契約・請求情報、個人情報を含まない通信」（クライアント→当社）だけ。業務データは境界を越えない。当社の不可触性は**構造的**（自己ホストで到達経路がない）であり、暗号化に依存しない。

### 1.3 データ境界の確認

| データ | クライアント側 | ホスト側 |
| --- | --- | --- |
| 業務データ（会計・名簿・ファイル・議事録） | 保持（自身のD1/KV/R2） | **保持しない** |
| 会員PII | 保持（暗号化） | **保持しない** |
| APIキー（Gemini/LINE/Claude） | 保持（暗号化） | **保持しない** |
| ライセンス／エンタイトルメント | トークンを保持 | ID・状態を保持 |
| B2B顧客情報（団体名・代表者・請求） | — | 保持（暗号化） |
| テレメトリ（バージョン・エラー数、PIIなし） | 送信元 | 受信・集計 |
| ホスト通知（お知らせ・更新、PIIなし） | 受信・表示 | 発行・配信 |

### 1.4 ポータブルコア＋パーツ／UIカスタマイズ（移植性アーキ）

詳細は [baku-office_portable-core_architecture.md](baku-office_portable-core_architecture.md)。要点：

- **コア能力 Port**：`SqlStore`／`StoragePort`／`AiPort`／`AgentPort`／`IdentityPort`。`middleware` が環境別アダプタを束ねた `ctx` を注入し、コア・パーツは `ctx.db/storage/ai/agent/identity` 経由でのみ動く（環境非依存）。
- **環境アダプタ／Profile**：CF（D1/KV/R2／クラウドLLM）が標準。AI は `ChatModel`（Gemini／Claude／**ローカルLLM(OpenAI互換)**）を実行時切替。`Profile` 検出で「フルクラウド〜オフライン」を可視化。ランタイム実体差し替え（D1↔SQLite・Workers↔Node）は deploy 時構成。
- **パーツ**：業務機能＝道具(`agentTools`)＋ナビ(`menu`)＋データ操作を `Part` として登録。`requiredRole` で認可。団体ごとに**有効/無効を選択**（管理画面の高度なオプション）。
- **UIカスタマイズ（共通ベース＋上書き・3層）**：①テーマ（ブランド名/ロゴ/配色・実行時）②ナビ構成（表示/ラベル/並び・実行時）③画面・部品の上書き（`src/overrides/` の Slot 注入／pages 同梱・配布時）。
- **移植性の保証**：適合性テスト（Node + `node:sqlite`）で同じコア/パーツを CF 非依存で実行・検証。
- 「専用ツール（団体X）＝ 環境Profile × 有効パーツ集合 × 連動する外部API」。3軸は直交する。

## 2. 料金プランと課金

### 2.1 プラン

| プラン（現行） | 旧名 | 内容 | AI | エージェント |
| --- | --- | --- | --- | --- |
| Free | X | 完全無料プラン | なし | なし |
| Plus | Y | AI利用・高度なオプション | Gemini（無料枠）／Claude（任意・要キー） | なし |
| Pro | Z | エージェント利用プラン | 同上 | LINE（標準）＋ローカルLLM（任意・オフライン） |

> 内部キーは **free / plus / pro** に統一（旧 X / Y / Z）。本書で以降に現れる「X / Y / Z」「free / Y / Z」は、それぞれ **Free / Plus / Pro** と読み替える。AIチャット・高度なオプションは Plus 以上、LINEエージェントは Pro。Claude は Pro 限定ではなく Plus 以上で任意選択可。

### 2.2 決済手段と有効化

- **申込時はプランを選ばない。全員 Free で開始**（`/apply` は団体情報のみ・Stripe 不要・ライセンス即発行）。
- **アップグレードは導入後にクライアント管理画面（`/billing`）から**：Plus / Pro を選んで Stripe決済。カードは即時確認・即時開放。**口座振込・コンビニ払いは年額をベース**とし、入金確認まで時間差が生じる。
- ダッシュボードからいつでもアップグレード可能（Free→Plus→Pro）。ダウングレード/解約時は機能のみ停止（データは保持）。
- 日本のStripeはカード・コンビニ・銀行振込（仮想口座での自動消込）に対応。日本の口座振替（自動引き落とし）には非対応（§付録B）。

### 2.3 プロビジョナル有効化

入金確認までの待ち時間で利用を止めない一方、未入金で有料機能を渡さないためのルール。

- Y・Z申込 → Stripe決済へ。**入金が確認されるまでは「無料プラン（X）相当」で稼働**（有料機能は開放しない）。
- カード：即時確認 → 即時にY/Z機能を開放。
- 振込・コンビニ（年額）：入金消込まではX相当 → **入金確認をもってY/Z機能を開放**。
- 仕組み：ライセンスは「ライセンスID＋エンタイトルメント状態（free/Y/Z）」を持ち、Worker は実行時にエンタイトルメントを確認して機能を分岐（§7.1）。Stripeの入金確認（Stripe → 当社へのWebhook）でエンタイトルメントを切り替えると、次回のWorkerの確認時にY/Z機能が開放される。

### 2.4 後からのアップグレード（追加課金）の挙動

- **起点**：ダッシュボードの「プラン・課金」からアップグレードを選択 → ライセンスIDを伴ってStripeへ遷移。
- **反映タイミング**：カードは即時確認→Webhookで即時に新プランへ切替。銀行振込・コンビニ（年額）は入金確認まで現プランのまま → 入金確認で切替。
- **期中アップグレード**：分かりやすさ優先で「入金確認時に切替・以後は新プランの年額期間を開始」を基本（カードはStripeのプロレーション併用も可）。
- **ダウングレード／解約**：現在の支払い済み期間の終了時に反映（期間内は上位機能を維持）。期末に無料（X）へ復帰。前払い分の返金なし。
- **更新（年額）**：カードは自動更新。振込・コンビニは更新時に請求書を発行して再入金。
- **未入金・更新失敗**：猶予期間（ダッシュボードに通知）→ 解消されなければ無料（X）へ復帰。**データは削除・ロックしない**。
- **Claude（Z上位）**：プラン変更（X/Y/Z）はStripe課金だが、Z内のClaude有効化は「連携設定でAnthropicキーを登録する」操作であり課金イベントではない（Claude利用料はクライアントがAnthropicへ直接＝BYOK）。
- **不変条件**：有料機能は入金が確認・継続している間のみ。無料（X）は常に使用可能。データはロックも削除もしない。

## 3. 配備：Deploy to Cloudflare ボタン

クライアント側セットアップ（§5の共通C）の中核。当社の雛形を、クライアントが自分のCloudflareアカウントへワンクリックで展開する仕組みで、当社はクライアントのアカウントに入らない。

### 3.1 前提：公開リポジトリには「難読化バンドル」を置く

Deployボタンは**公開のGitHub/GitLabリポジトリ**を対象とする（セルフホスト版は非対象、Workersのみ対象でPagesは非対象）。当社は**TypeScriptソースを非公開リポジトリに置き、CIで難読化したバンドル＋`wrangler.jsonc`＋D1マイグレーションだけを公開リポジトリに配布**する。クライアントが目にするのは難読化済みコードのみ。

### 3.2 クリック後の処理の流れ

1. クライアントがDeployリンクを開き、Cloudflareにサインイン（§5のとおりGoogleで作成・ログイン）。
2. Cloudflareが当社の公開リポジトリを**クライアント自身のGitHub/GitLabアカウントに複製**（新規リポジトリ作成）。以後クライアントはこの複製を起点に運用・更新。
3. Cloudflareが複製先の `wrangler.jsonc` を読み、必要なリソース（D1・KV、R2は§11の高度オプション時）を**クライアントのアカウント上に自動作成しWorkerにバインド**。
4. **セットアップ画面**でWorker名などを確認。**認証キーの入力はさせない**（ライセンスは初回起動時に自動取得＝§4）。
5. **D1マイグレーション**は `package.json` のビルド／デプロイスクリプトに記述すればCloudflareが自動検出して反映。
6. **Workers Builds（CI/CD）が設定**され、本番ブランチへのプッシュごとに自動ビルド・デプロイ。`*.workers.dev` 等でアプリ起動。

### 3.3 更新（バージョンアップ）

当社が公開リポジトリに新バージョンを出すと、クライアントは**自分の複製（フォーク）を上流と同期**するだけでWorkers Buildsが自動再デプロイ。発火操作はすべてクライアント側で行われ、当社は介在しない。

### 3.4 単一Worker構成

管理画面（静的アセット）とWEBアプリ（API／ロジック）は**1つのWorkerに同居**（Workersは静的アセット配信に対応）。デプロイ単位が1つになり、セットアップが簡単になる。

## 4. ライセンスの自動アクティベーション（認証キー入力なし）

クライアントに認証キーを貼り付けさせない。初回起動時に、申込時と同じGoogleアカウントで認証してライセンスを自動取得する。

1. クライアントがDeploy後に初めてアプリを開くと、ライセンス未保持を検知し**自動で当社のアクティベーション画面へリダイレクト**。
2. クライアントが**申込時と同じGoogleアカウントでログイン**（Google認証は当社側で処理。クライアントのアプリにGoogle認証実装は不要）。
3. 当社が認証済みGoogle IDと申込時ライセンスを突き合わせ、**短命の認可コード**をアプリのコールバックURLへ返す。
4. アプリ（クライアントのWorker）が裏でコードを当社と交換し、**署名済みライセンストークンを受け取ってKV/D1に保存**。以後は保存済みトークンで動作。

エンタイトルメント更新は§13.1の確認通信に相乗り（PIIなし）。不正アクティベーションはGoogleログインがゲートになるため防げる。標準的なOAuth2の認可コードフローで実現でき、特別なCloudflare機能には依存しない。

> セキュリティ実装：上記3の認可は**ホストが Ed25519 で署名した relay エンベロープ**（`{sub,email,name,exp}`）として返り、クライアントは公開鍵で検証する。4のトークン発行（`/api/activate-by-email`）も**その署名 relay を必須**とし、ホスト自身が再検証する（生メール直叩きでの発行を遮断・期限切れ拒否）。

> ライセンス（当社発行・初回に自動取得）と、ユーザー供給APIキー（§10.3、実行時に管理画面から変更可）は、取得経路も保存・更新のタイミングも別物として扱う。

## 5. オンボーディング全体フロー

### ホスト側（申込〜ライセンス発行）

1. Googleアカウントを用意（**団体専用の共有アカウント推奨**。引継ぎが容易になる）。
2. 当社の申込画面にGoogleログイン。
3. 団体情報を入力。
4. プランを選択（X／Y／Z）。
5. 各種同意事項を提示・同意。
6. 分岐：**X** → その場でライセンスを即発行（Stripeなし）。**Y・Z** → Stripe決済画面へ（入金確認まではX相当で稼働）。
7. クライアントへ、初期セットアップページと**個別のDeployリンク**を送付（認証キーの送付は不要）。

### クライアント側 初期設定（共通：全プラン）

- **A. Cloudflareアカウント作成**（「Sign in with Google」、団体専用Googleで）。
- **B. GitHubアカウント作成**（「Continue with Google」、同じ団体専用Googleで）。
- **C. Deployボタンで展開**：個別Deployリンクを開く → CF認可 → リポジトリ複製・D1/KV自動作成・ビルド／デプロイ（単一Worker）。初回起動時に申込時のGoogleで自動アクティベート（§4）。

→ ここまでで、クライアントは自分のCloudflare上で動くWEBアプリを**完全無料**で起動できる（Xプランの完成形）。

### Yプラン追加（AI利用）

- **D. Gemini APIキーの取得**（Googleアカウントのみ、無料）。
- **E. 管理画面でキーを入力・保存**（保存時バリデーション → 暗号化してKV/D1へ、§10.3）。

### Zプラン追加（エージェント利用）

- **F. 公式LINEを作成**（既にあればスキップ）。
- **G. LINE Developers Console でチャネル作成**、シークレットとアクセストークンを取得。Zでは**Webログイン用「LINE Login」とエージェント用「Messaging API」の2チャネル**を用意する（§6.2）。
- **H. 管理画面でキーを入力・保存**（同上）。
- **（上位・分離オプション）Claude**：高度な知能や機微情報の取り扱いを希望する場合のみ、別途 Anthropic APIキーを管理画面で設定（BYOK）。

## 6. 認証とマルチユーザー（組織／個人）

### 6.1 二つのコンテキスト

| コンテキスト | 対象プラン | ログイン | 範囲 |
| --- | --- | --- | --- |
| 組織コンテキスト | 全プラン（X/Y/Z） | 組織のGoogleアカウントのみ | 会計・ファイル・人／ロール管理・組織ナレッジ |
| 個人コンテキスト | Zプランのみ | 個人のLINE／Discord／ID・PASS | 自分の領収書・メモ・タスク・予定（共有設定付き）＋付与ロールに応じた組織セクションへのアクセス（§6.4） |

無料・Yプランは組織コンテキストのみ。すべて組織のCFアカウント内で完結する。

### 6.2 認証モデル

- **組織**：設定時に作った組織Googleアカウント（共有・最上位）。
- **個人**：LINEログイン（LINE Login チャネル。初回に氏名・役職を登録し LINE userId と個人レコードを紐づけ）／LINE非連携の場合は Discord（OAuth）または ローカルID・PASS。認証フローは組織のWorkerで実行し、レコードは組織のD1に保存。
- LINEは用途別にチャネルが分かれる：Webログイン用「LINE Login」とエージェント用「Messaging API」。Zで両方使う場合は両チャネルのキーを連携設定に登録（契約案内で明示）。**前提**：エージェントの会員照合は LINE Login の `sub` と Messaging API の `userId` が同一プロバイダで一致すること（別プロバイダだと会員判定不可）。
- **実装**：認証は Identity Port（`core/identity.ts`）経由。ローカルPASSは **PBKDF2**（塩・ストレッチ・定数時間比較・旧SHA-256はログイン時に透過再ハッシュ）。OAuth state／セッション／一時 `pending_oauth` は **HMAC 署名**で改竄を防止。

### 6.3 参加ゲート（招待コード＋承認制）

1. 組織管理者（Googleログイン）が人管理画面から招待コードを発行。**有効期限1週間・使用1回のみ**、付与する既定ロールを指定。
2. 個人は招待コードで個人ログイン（LINE／Discord／ID・PASS）し、初回に氏名・役職を登録。
3. 登録されると**組織のメールアドレスへ通知**が届き、状態は**承認待ち（pending）**。組織管理者が**管理画面から承認**して初めてactiveになる。
4. 招待コードが漏れても管理者承認なしには参加できず（二重ゲート）、管理者は却下・無効化も可能。

### 6.4 ロールと権限（ロール別の組織アクセス）

組織のGoogleログインは最上位（全権）。各個人にはロールを付与し、**個人コンテキストからも、付与ロールに対応する組織セクションにアクセスできる**（例：会計担当は経理セクションを開けるが、他は開けない）。

- **管理者（代表・副代表）**：全セクション＋人・招待・承認・直接DB操作・プラン課金。
- **会計**：経理関係のみ（会計・会計系の共有承認）。
- **庶務**：ドキュメント関係のみ（ファイル・資料・議事録・スケジュール・ナレッジと、その共有承認）。
- **その他（任意設定）**：管理者が個別にセクション権限を割り当て。
- ロール未付与の一般メンバー：個人アイテムのみ。

権限表（既定。「その他」は任意設定、管理者が調整可）：

| セクション | 組織Google | 管理者 | 会計 | 庶務 | 一般 |
| --- | --- | --- | --- | --- | --- |
| 会計（経理） | フル | フル | R/W＋承認 | なし | なし |

### 6.5 個人コンテキストの位置づけと組織優位

個人コンテキストは「組織の中で個人が使う作業領域」であり、私的用途は想定しない。よって個人間の秘匿は提供せず、**組織を上位とする構成**を採る。

- **per-user 暗号は採用しない。** 個人アイテムの保護は§10の暗号化＋アプリ層の論理アクセス制御＋操作監査ログで行う（ユーザー間の暗号的分離ではない）。
- **組織管理者のオーバーサイト**：データは組織に帰属し、組織管理者は個人アイテムを含め閲覧・管理・エクスポートできる。`share_scope=personal` は「他メンバーの業務ビューに能動的に出さない」を意味するに留まる。
- 私的内容を置かないことは各個人の責任（規約・初回案内で明示）。

## 9. 共有承認ワークフロー（個人→組織）

個人アイテムを `share_scope=org` にした場合、即座に組織データへ反映せず、**GitHubのプルリクエストのように一度組織側の共有プールに「承認待ち」で入る**。担当ロール（または管理者）が確認し、承認で組織の共有DB／ストレージへ反映、却下で理由を添えて差し戻す。領収書・スケジュール・資料・メモなど全種別を同じ流れで扱う。

- 状態遷移：`personal` →（共有=組織に設定）→ `pending`（共有プールで承認待ち）→ `approved`（組織へ反映）／ `rejected`（差し戻し）。`personal_items.review_status` で管理。
- 承認担当の振り分け：会計系（領収書等）→ 会計または管理者、文書・資料・スケジュール・メモ → 庶務または管理者（§6.4）。
- **反映時の扱い**：承認済みの共有領収書は**会計取引のドラフトを自動生成**（金額・日付・店名→摘要、科目は履歴・ルールから推定）。担当はレビュー画面で**承認前に編集でき、承認後も修正可能**。文書・資料は組織の共有ストレージへ。
- UI（仮・GitHubのプルリクエスト風）：「共有承認キュー」を一覧表示（種別アイコン・概要・提出者・日時・担当ロールのタグ・状態）、行から確認するとレビュー画面（領収書は画像プレビュー＋自動生成された取引ドラフト＝編集可、メモ・資料は本文プレビュー）に入り、承認（反映）／却下（理由付き）で処理。担当ロールの個人コンテキスト内の該当セクションにも同じキューを出す。

## 10. 暗号化とデータ分類

### 10.1 暗号化モデル

**マスター鍵を1本、Worker Secret に置き、API鍵と会員PIIを共通方式で暗号化する。**

- マスター鍵：`MASTER_KEY`（per-deployment、AES-256相当の乱数）。**Worker Secret に保持（推奨）**。鍵の取得・保管は **KvPort 経由に分離**（`masterKeyCtx`／§移植性アーキ 14-3）＝Profile で保管先を差し替え可能。
  - secret 未投入時は zero-config 維持のため KV に自動生成するが、**鍵が暗号文と同居**し下記の前提を崩すため、**重大診断を記録**し管理画面（診断）に警告表示する。本番は `wrangler secret put MASTER_KEY` を投入する。
- アルゴリズム：AES-256-GCM（WebCrypto `crypto.subtle`）。レコードごとに乱数12バイトIV、保存形式 `IV ‖ 暗号文 ‖ 認証タグ`（base64）。
- ドメイン分離（任意・推奨）：HKDFで用途別サブ鍵（`api-keys`/`member-pii`/`files`）を導出。鍵は1本のまま用途間を分離。
- ローテーション：新 `MASTER_KEY` 設定 → 全レコード再暗号化（保守作業。常用不要）。
- **守るもの／守らないもの**：Cloudflareは保存時暗号化済み。本アプリ層暗号化は「クライアント自身のストレージの部分漏れ」（エクスポート・バックアップの取り扱いミス、値だけ漏れるバグ、サブプロセッサ事故）に効く。**アカウントの完全侵害には無力**。効く条件は**マスター鍵を暗号文（KV/D1）と別の漏洩クラス（Worker Secret）に置くこと**。鍵を暗号文と同じKV/D1に置くと単なる難読化に成り下がる。

### 10.2 暗号化対象データ分類表

原則：暗号化した項目はサーバ側で検索・集計できなくなる。よって「集計が必要な会計値は平文／個人情報は暗号化」を基本とする。会員名簿は小規模なので表示時にまとめて復号でき、PII暗号化と一覧表示は両立。凡例 — ✔必須／△推奨／✗不要（平文）。

| 区分 | 例 | 機微度 | 暗号化 | 保存先 | 検索・集計 | 備考 |
| --- | --- | --- | --- | --- | --- | --- |
| APIキー | Gemini／LINE／Claude | 高 | ✔ | KV または D1 | 不要 | 保存時バリデーション・マスク表示 |
| ライセンストークン | 当社発行の署名トークン | 高 | ✔ | KV／D1 | 不要 | 初回アクティベートで取得、定期更新 |
| 会員PII | 氏名・連絡先・住所 | 高 | ✔ | D1 | 表示時に復号 | 第三者の個人情報。最重要保護対象 |
| 会計の数値・属性 | 金額・日付・科目・口座区分 | 低 | ✗ | D1 | する | 帳票・グラフのため平文 |
| 会計の摘要 | 取引メモ（自由記述） | 中 | △ | D1 | 原則しない | 氏名等が入りうる |
| ファイル本体 | アップロード文書 | 中〜高 | △ | R2 | 不要 | PII含むなら本体も暗号化 |
| ファイル／予定のメタ | 名前・サイズ・日時 | 中 | ✗（名前注意） | D1／R2メタ | 一覧表示 | 名前にPIIを入れない運用 |
| スケジュール／議事録 本文 | 予定詳細・議事メモ | 中 | △ | D1／R2 | 原則しない | 氏名・議論が入りうる |
| 操作・エラーログ | 工程ステータス・例外 | 低 | ✗ | D1／ログ | する | PIIをログに残さない |
| ヘルステレメトリ | バージョン・エラー件数 | 低 | ✗ | 当社へ送信 | 集計 | 境界を越える唯一のクライアント発データ |
| 団体設定 | 団体名・科目マスタ・会計期 | 低 | ✗ | D1 | する | 団体名は組織名 |
| 決済資格情報 | カード番号・口座番号 | — | **保存しない** | — | — | 決済事業者（Stripe等）側に保持 |

### 10.3 APIキーの設定（CF画面を開かずに管理画面から）

Gemini・LINE等のAPIキーは、**Worker Secretsではなく、クライアント自身のKV／D1に、管理画面フォームから保存**する（CFダッシュボード不要・CF APIトークン不要・再デプロイ不要）。流れ：フォーム入力 → テスト呼び出しで保存時バリデーション → §10.1の `MASTER_KEY` でアプリ層暗号化して保存 → 実行時に復号し、キーの有無・有効性で機能分岐（§7.1）。

## 11. ストレージ構成（R2の高度なオプション）

ファイル保存（R2）はクレジットカード（またはPayPal）登録が前提（無料枠でも有効化に必要、無料枠を超えなければ課金なし）。任意団体はカードを持たないことが多いため2モードで設計する。

- **標準モード（R2なし・カード不要）**：基本配備はD1＋KVのみ。ファイルはフォールバックでKVに保存し、1ファイルの上限はバックエンドの物理上限に従う（KV値上限25MiB、D1にBLOBで持つ場合は2MB）。推奨の既定上限は安全側に**5MB**、高度なオプションから**25MiBまで変更可**。KVは無料枠の制限（1日1,000書き込み・合計1GB）があるため、標準モードのファイル機能は軽量用途向け。
- **高度モード（R2あり・カード必要）**：「高度なオプション」からR2を有効化。領収書画像・文書など本格的なファイル保存をR2で行う。

R2の有効化は**クライアントがCloudflareを直接設定する数少ない箇所**（カード登録 → R2バケット作成 → バインド。手順は当社が案内）。プログラムは**R2バインディングの有無を実行時に検出**し、あれば利用、なければ標準モードにフォールバック。R2有効時はアプリ側で上限を設けない（上記の上限はフォールバック時のみ適用）。R2は使用量ベース課金のため、支払い方法が無効化され30日以内に更新されないとR2データが削除され得る点を案内する（標準モードは影響を受けない）。

## 12. 直接DB／ストレージ操作（WEB UI・自己責任）

クライアントが管理画面から自組織のD1／R2を直接操作できる（ファイルのアップロード・削除、レコードの編集・削除）。自己責任とするが、誤操作による不可逆な損失を避けるため、機能を妨げない範囲で既定の安全策を組み込む。

- **ソフトデリート＋ゴミ箱**：削除は一旦ゴミ箱へ（**30日**保持で復元可）。30日経過後または明示操作で完全削除。
- **破壊的操作の確認**：削除・一括操作は確認ダイアログ、危険／一括操作はtype-to-confirm。
- **操作監査ログ**：誰が・いつ・何を、を組織のD1内に記録（`audit_log`）。
- **ロールによる権限ゲート**：直接DB／ストレージ操作は組織Google／管理者のみ（§6.4）。
- **一括破壊前のエクスポート促し**：CSV／ZIPのバックアップを促す。

完全削除（ハードデリート）即時も設定可能だが既定はソフトデリート推奨。レコード操作は表形式の構造化UI（生SQLは既定で提供しない）。すべて組織のCFアカウント内で完結し、エージェント経由でも同一の内部APIを通る。

## 13. 通知・サポート・可観測性

### 13.1 ホスト通知（ポーリングと表示優先度）

当社→クライアントの一方向プル型。クライアントのWorkerが**統合チェックエンドポイント**を取得し `{ entitlement, latestVersion, notices[] }` をまとめて受け取る（アクティベーション§4・エンタイトルメント確認に相乗り、PIIなし）。各通知は `id・severity・本文・next_poll_hint` を持ち、再取得間隔はサーバ応答の推奨値に従う。既読・解除状態はKV/D1に通知IDごとに保持。

| 種別 | 重要度 | 取得契機・間隔 | 表示 | 再通知 |
| --- | --- | --- | --- | --- |
| システム更新（新バージョン） | info | 起動時＋6〜12hごと | 常設バナー＋一覧 | 更新（フォーク同期）まで継続 |
| メンテ・障害情報 | important | 起動時＋1〜3hごと | 解除可バナー＋一覧 | 既読で解除、未解決中は再掲 |
| 重要なお知らせ（規約変更等） | critical | 起動時＋数時間ごと＋メール併用 | 確認モーダル（要承認）＋一覧 | 承認まで毎回提示 |
| 一般のお知らせ | info | 起動時＋日次 | 一覧（未読バッジ） | 既読で消える |

プル型は即時性が弱いため、緊急のメンテ周知は当社保有のB2B連絡先メールを併用する。

### 13.2 ヘルステレメトリ（オプトイン）

クライアントの同意のもと、**バージョン番号とエラー件数の集計のみ**（PIIなし）を当社へ送信。当社からクライアントのアカウント内部は見えないため、この任意テレメトリが故障把握の主な手段になる。

### 13.3 構造化ログ・診断

オンボーディング各ステップ（§5のA〜H）と実行時処理で、失敗時に原因を特定できる**構造化エラーログ**を出力（PIIを含めない）。クライアント自身の環境に保持し、必要時に診断情報としてエクスポート・共有。

## 14. セキュリティ・データ・バックアップ

- 業務データはクライアント自身のCloudflare内に保管。当社はアクセスしない（§1.2）。
- 当社が把握するのは契約・請求に必要なB2B顧客情報のみ。通信は暗号化。APIキー・ライセンスは暗号化して保管（§10）。

- Z（特にClaude）で会員の機微情報をAIに渡す場合は、有料枠の利用や、団体メンバーへの開示・同意の取得（雛形は当社が提供）を前提とする。

## 付録A：暗号化の実装スケッチ（WebCrypto / 擬似コード）

```
key = importKey(base64Decode(env.MASTER_KEY), "AES-GCM")        // 用途別はHKDFで派生
iv  = randomBytes(12)
ct  = AES-GCM.encrypt(key, iv, plaintext)                       // tagはctに付随
store = base64( iv ‖ ct )                                       // KV/D1へ

// 復号
iv, ct = split(base64Decode(store))
plaintext = AES-GCM.decrypt(key, iv, ct)
```

保存時バリデーション：`復号 → 対象API（Gemini/LINE）へテスト呼び出し → 成功なら保存し ✓、失敗なら ✗ とエラー詳細を表示`。

## 付録B：制約・要確認事項

制約：
- Deployボタンは公開 github.com／gitlab.com のリポジトリのみ、Workersのみ（Pagesは非対象）。
- 日本のStripeは口座振替（自動引き落とし）に非対応（カード・コンビニ・銀行振込は対応）。口座振替を将来加える場合はStripe外サービス（ゆうちょ・信金・JA対応を要確認）。

- R2はカード登録が前提。

着手前に確認：
- `MASTER_KEY` を「無ければ生成」で実行時に設定する方法（Workers Builds工程／デプロイAPIでの実行時Secret設定の可否）。難しければデプロイのセットアップ画面で乱数を1回だけ設定する方式に切替。
- Deployボタンの環境変数／シークレット引き渡しの最新仕様、Workers静的アセット（Pages後継）の最新仕様、各決済の最新料率。
- 「△推奨」項目（摘要・議事録等）の暗号化範囲、ホスト側B2B顧客PIIの暗号化対象フィールド。
- 「その他」ロールの権限割り当てUI細部、共有承認キューの一括承認の可否、取引ドラフトの科目推定ロジック。
- R2有効化の案内手順（カード登録→バケット作成→バインド）。
- エージェント（既存実装）と内部APIのマッピング詳細。
- 法務面（クラウド例外の当てはめ、資金フロー、個人情報の取扱い）は最新法を前提に顧問へ確認。
